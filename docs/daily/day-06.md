- `terraform -chdir=terraform apply` works from any fresh terminal (winget dir on PATH).
- **Bronze must already hold ‚â•1 hour** for Spark to read. The daily `terraform destroy` empties the bucket, so re-ingest one hour first (run `python run.py`, or the `ingest_hour` step) before submitting the Spark job.

## Focus
Write a **PySpark** silver transform that reads `gs://‚Ä¶/date=‚Ä¶/hour=‚Ä¶/‚Ä¶json.gz` from bronze with an **explicit schema**, flattens the GH Archive envelope (`id, type, actor{id,login}, repo{id,name}, created_at, ‚Ä¶`), **dedupes on `id`**, casts `created_at` ‚Üí `TIMESTAMP`, and writes **date/hour-partitioned Parquet** to silver with a **partition-scoped overwrite** (idempotent, mirroring Day 5). Run it single-node in Docker via `spark-submit`; prove the output matches the Day 5 per-type counts and that a re-run replaces ‚Äî not duplicates ‚Äî a partition.

## Build steps
Execute in order; one small commit per step. Steps 1 and 6‚Äì7 are setup/verify, not commits.

1. **Recreate infra + land one bronze hour.** Day 5 tore the cloud down. `terraform -chdir=terraform apply`; confirm `.env` + ADC. Then **put one hour in bronze** (`python run.py` lands `2024-01-01 15:00`, or just the ingest half) so Spark has input ‚Äî confirm the object exists in GCS. *(Setup, not a commit.)*

2. **Add a silver landing location (Terraform).** Silver Parquet needs a home. *Decision:* add a **dedicated `silver` GCS bucket** in `terraform/main.tf` (mirrors `bronze`, but **no object versioning** ‚Äî silver is fully reproducible from bronze, so versioning is wasted cost; keep `public_access_prevention = "enforced"` + `force_destroy = true` for dev teardown). *(Rejected alt: a `silver/` prefix inside the bronze bucket ‚Äî simpler, no new resource, but it mixes raw + derived data under one IAM/lifecycle boundary; separate buckets per medallion layer is the production-correct shape and gives clean per-layer lifecycle rules.)* **Note the IAM gap:** the pipeline SA's `storage.objectAdmin` is scoped to **bronze only** ‚Äî a silver bucket needs its own grant. We're still running as **personal ADC** today (SA impersonation remains deferred on the security backlog), so this isn't blocking now; add the SA‚Üísilver grant when Day 7/the SA switch lands. *Commit.*

3. **Stand up Spark in Docker (single-node) + reach GCS.** Create `spark/` with a small Docker setup (a `Dockerfile` on an `apache/spark` / `bitnami/spark` base, or a compose service) running **`local[*]`** ‚Äî *not* a real cluster (single-node is intentional; see constraints). The hard part is **reading `gs://` from Spark**: Spark's Hadoop layer needs the **GCS connector jar** (`gcs-connector-hadoop3-*-shaded.jar`) on the classpath and `fs.gs.*` configured to use **ADC** (`spark.hadoop.fs.gs.auth.type=APPLICATION_DEFAULT`, or point the connector at the bind-mounted ADC file). Mount the ADC `:ro` and the `spark/` job code in, same as Airflow. **Verify** with a throwaway `spark-submit` that just reads the bronze gz and prints `.count()` ‚Äî get *connectivity* green before any transform logic. *Commit (the Spark Docker setup).*

4. **Write the silver transform (explicit schema ‚Üí flatten ‚Üí dedupe ‚Üí type).** `spark/silver_events.py`, structured as a **pure DataFrame‚ÜíDataFrame function** for testability:
   - **Explicit `StructType`**, *not* `inferSchema` ‚Äî inference triggers an extra full pass over the data, is non-deterministic across files, and isn't a contract. Define the envelope (`id`, `type`, `actor: struct<id,login>`, `repo: struct<id,name>`, `created_at`, `public`); let varying `payload` ride along as a raw/string column for now (per-event-type payload explosion across the ~15 types is iterative ‚Äî don't model all 15 today).
   - **Flatten** to columns: `event_id`, `event_type`, `actor_id`, `actor_login`, `repo_id`, `repo_name`, `created_at` (‚Üí `to_timestamp`), plus `event_date` / `event_hour` partition columns derived from `created_at`.
   - **Dedupe** with `dropDuplicates(["event_id"])` ‚Äî GH Archive repeats events across adjacent hourly files; the `Counter` never deduped, so this is a *real* correctness upgrade. (Spark's JSON reader also sidesteps the Day 3 `splitlines()` NDJSON gotcha entirely ‚Äî another logged reason to move off the hand-rolled parser.)
   - *Commit (the transform function).*

5. **Write partitioned Parquet to silver, idempotently.** `df.write.partitionBy("event_date","event_hour").parquet(silver_path)` with **`mode="overwrite"` + `spark.sql.sources.partitionOverwriteMode=dynamic`** ‚Äî so re-running one hour replaces **only that hour's partition**, never the whole silver dataset. This is the **direct analog of Day 5's partition-decorator `WRITE_TRUNCATE`**: same idempotency rule (partition grain = load grain), enforced in the lake instead of the warehouse. *(Default `overwrite` is `static` and would nuke the entire silver directory ‚Äî the exact trap Day 5 warned about, one layer up.)* *Commit.*

6. **Run + verify (counts, dedupe, idempotency).** `spark-submit` the job for `2024-01-01 15:00`. Read the Parquet back (Spark or pandas/pyarrow) and: (a) **`groupBy("event_type").count()` should match the Day 5 `hourly_event_counts`** for that hour ‚Äî *modulo* dedupe (silver may be slightly lower if that hour had cross-file dupes; explain any delta, don't hand-wave it); (b) confirm `event_date=‚Ä¶/event_hour=‚Ä¶/` partition dirs exist; (c) **re-submit the same hour and confirm the partition is replaced, not doubled** (row count steady) while sibling partitions are untouched. *(Setup/verify, not a commit.)*

7. **(First real unit test) + cost hygiene + status.** `tests/` is empty; CLAUDE conventions require pytest on non-trivial transform logic ‚Äî and this is the first. Add `tests/test_silver_events.py`: build a tiny in-memory DataFrame with a duplicate `id` and a malformed `created_at`, run the pure transform, assert dedupe + the cast behave. *(Small commit.)* Then **`docker compose down`** (Spark + any Airflow); **`terraform -chdir=terraform destroy`**. Update `CLAUDE.md` **Current status / Next up** (‚Üí Day 7: load silver Parquet ‚Üí BigQuery + wire Spark into Airflow, retire the `Counter` + `max_active_runs=1`); log the Spark decisions in `decisions.md`; check off the done criteria. *(Leave the Phase 1 PySpark box **unticked** ‚Äî it isn't done until the job is wired into Airflow on Day 7.)*

## Constraints & conventions
- üß± **Single-node Spark is intentional ‚Äî and you must be able to say what changes at scale.** `local[*]` in one container is *not* a cluster; the **code is identical** to a real one. Be ready to articulate: gzip is **non-splittable** (one `.gz` = one task, so one hour = one partition ‚Äî fine now; at scale you'd `repartition` after read, and the splittable columnar Parquet you *write* fixes it downstream), `spark.sql.shuffle.partitions` tuning, executor memory, and why you'd move to **Dataproc/EMR/Databricks**. That articulation is worth as much as the code (per the blueprint).
- üéØ **Partition grain = write grain, again.** Dynamic partition overwrite replaces a whole partition; partitioning Parquet by `event_date`/`event_hour` is what keeps "replace this hour" idempotent ‚Äî the **same rule as Day 5's `$YYYYMMDDHH` decorator**, one layer up in the lake. Never use a static `overwrite` over the silver root.
- üîÅ **Idempotency is mandatory (hard rule).** Re-running the Spark job for an hour must replace that hour's Parquet, not append/duplicate. Prove it (Step 6), don't assume it. Bronze read is read-only; bronze's `blob.exists()` skip is unchanged.
- üìê **Explicit schema, never `inferSchema`.** A schema is a *contract* that fails loudly on drift; inference is an extra data pass that silently redefines it (same principle as the Day 3 `mode="REQUIRED"` BQ schema). Decide the corrupt-record policy deliberately (`PERMISSIVE` + a `_corrupt_record` column, or `FAILFAST`) ‚Äî pipelines fail loudly on bad data.
- üßµ **Don't boil the ocean on event types.** Flatten the common envelope + dedupe + type today; full normalization of all ~15 `payload` shapes is iterative (and partly a Phase 2 dbt-staging concern). Land a correct, narrow silver table first.
- üîê **Creds: ADC bind-mounted `:ro`, no keys in the image.** Same pattern as Airflow ‚Äî nothing `COPY`d in. Still **personal ADC**, not the pipeline SA (impersonation/workload-identity switch + the silver-bucket SA grant remain on the security backlog). No secrets in git; verify Spark's `spark/work`/`derby.log`/`metastore_db` scratch and any `_SUCCESS`/local Parquet are gitignored.
- üîè **PII watch (carries from Phase 0).** GH Archive `payload` carries **author emails** (e.g. `PushEvent` commits). We're keeping `payload` as a raw column today and **not** surfacing emails as typed columns ‚Äî hash/drop them when payload explosion lands in staging (Phase 2). Don't let an email become a first-class silver field by accident.
- üí∏ **Cost hygiene.** Spark runs **local in Docker** (free ‚Äî no Dataproc). Costs are the bronze read + tiny silver Parquet storage. Reads scan GCS, not BigQuery, so no bytes-billed today (the BQ load is Day 7). `docker compose down` + `terraform destroy` at session end.
- üì¶ **Small, reviewable commits & understand-don't-rubber-stamp.** One step per commit (e.g. `add silver GCS bucket`, `add single-node spark docker setup`, `add pyspark silver transform`, `write partitioned parquet to silver`). I want the *why* for explicit-schema-vs-infer, dynamic-vs-static partition overwrite, gzip non-splittability, and dedupe-on-`id` before each lands.

## Learning goals
1. **The Spark execution model** ‚Äî driver/executor, transformations vs. actions (lazy ‚Üí a job only runs on an action), partitions as the unit of parallelism, and why `local[*]` is the *same code* as a cluster (what genuinely changes at scale).
2. **Reading semi-structured JSON at scale** ‚Äî explicit `StructType` vs. `inferSchema` (cost + contract), the corrupt-record modes, and why the Spark reader sidesteps the hand-rolled NDJSON `split("\n")` parser.
3. **gzip non-splittability** ‚Äî why one `.gz` = one task, what that means for parallelism on a single large file, and how the splittable, columnar **Parquet** you write fixes it for every downstream reader.
4. **Dedupe + typing as the silver contract** ‚Äî `dropDuplicates` on the event key (a shuffle), `to_timestamp` casting, and why bronze‚Üísilver is where "raw and repeated" becomes "clean and typed."
5. **Partition-scoped idempotency in the lake** ‚Äî `partitionOverwriteMode=dynamic` as the lake analog of Day 5's BigQuery partition decorator: the same "replace exactly one partition" rule, enforced one layer up.
6. **Parquet as the silver format** ‚Äî columnar + compressed + splittable + schema-embedded, partitioned by `date/hour` for pruning (continuing the Hive-style layout from bronze).
