# Day 16 — Consume your own API: a Streamlit dashboard that never touches the warehouse

> **Phase 3 · Week 6 (🏁 the finish line).** Picks up from Day 15 (five sessions, 2026-07-12→17): `api/` serves four endpoints over gold — `/trending`, `/languages/momentum`, `/leaderboard` on billed query jobs (every value a bound `@param`, `maximum_bytes_billed=100 MB`, TTL-300 s cache with `X-Cache`) and `/runs` on the free `list_rows` path (zero query jobs, proven against job history). Suite **69** green. But the API has never had a consumer that wasn't `curl.exe` or `/docs`. Today the platform grows its last surface: a dashboard a non-engineer could look at — and the discipline that it reaches the warehouse **only through the API**.
>
> **Scope call (read first):** IN — a `dashboard/` package: an HTTP client module wrapping the four endpoints (timeouts, typed returns, transport errors translated into one domain error), a Streamlit app with a date picker and four panels (trending repos, language momentum, contributor leaderboard, pipeline status from `/runs`), explicit empty/NULL/error states, and a deliberate two-layer cache with a manual refresh. OUT — GitHub Actions CI (Day 17, the last Phase-3 item), auth + rate limiting + pinned deps (🔐 pre-deploy gate — both processes stay on localhost today), any deploy (Cloud Run / Streamlit Cloud — Phase-3 stretch), a `/repos/{id}` drill-down endpoint (the dashboard doesn't need one yet; add it when a panel does), and any new gold model (dbt is untouched today — **PASS=69** proves it).
>
> **The day's central decision, stated up front and honestly:** *does the dashboard call the API, or query BigQuery directly?* Today: **the API, always** — because everything that makes warehouse access safe lives in `api/queries.py` and `api/cache.py` (bound parameters, the 100 MB cap, the TTL, deterministic ORDER BY). A dashboard with its own `bigquery.Client` is a **second, uncontrolled path to the warehouse**: uncapped, unparameterized, uncached, and duplicating SQL that will drift from the API's. One door, guarded. **Rejected (and when each wins):** *Streamlit → BigQuery directly* wins when the dashboard is the **only** consumer and no API exists — one hop instead of two, no serialization tax, and nothing to keep in sync (the moment a second consumer appears, it loses). *Looker Studio / a BI tool on the gold marts* wins for business users who need self-serve exploration and scheduled delivery with **no code to maintain** — it is genuinely the right answer for a company, and it is the wrong answer here precisely because it would connect straight to BQ and skip the API, leaving nothing engineered to defend. *A React/Next SPA against the API* wins when you need real product UX, mobile, or public traffic — it costs a build toolchain and a JS stack that teaches nothing new about data engineering. The production-vs-simplified line, said loudly: **Streamlit is a single-process, single-user, server-rendered tool** — it re-runs the whole script top-to-bottom on every widget interaction, which is why caching is not optional, and it has no multi-tenancy, no auth, and no horizontal story. That's honest for one developer on localhost; the at-scale shape is a proper frontend against an authenticated, rate-limited API behind a CDN. Say "exercised, not stress-tested" and mean it.

- **Orientation — the one Streamlit idea that surprises people:** there is no event loop and no callbacks. Every interaction **re-executes `app.py` from line 1**. A naïve app therefore re-fetches every endpoint on every click. `@st.cache_data(ttl=…)` is what makes the re-run model survivable — it is load-bearing architecture here, not a micro-optimization. Read that sentence again before step 5.
- **Two commands you'll live in:** `python -m uvicorn api.main:app --reload` (repo root, main `.venv`) and `.\.venv-dashboard\Scripts\python.exe -m streamlit run dashboard/app.py` (repo root, dashboard venv). Watch the **uvicorn access log** — it is today's best debugging instrument, exactly as it was on Day 15 s3 when it caught the 710-star under-sum.

## Focus

1. **The API client — one door, one error type.** 🧑‍💻 New `dashboard/api_client.py`: a `BASE_URL` constant (`http://127.0.0.1:8000`, overridable by env var — the dashboard's only configuration), a `DashboardError(Exception)` carrying a human-readable message, and one private `_get(path: str, params: dict) -> dict` that calls `requests.get(..., timeout=…)` and raises `DashboardError` on **every** failure class: `ConnectionError` (API not running — the most common one all day), `Timeout`, `4xx`, `5xx`. Then four thin wrappers returning the parsed payload with type hints. The teaching, three parts: (a) **a request without a timeout hangs forever** — `requests` has no default timeout, and a UI thread blocked on a dead socket is an outage with no error message; pick one and comment *why* that number; (b) **translate at the boundary** — the UI must never see a `requests` exception, because the day a `requests`-shaped error reaches a render function, the UI has silently coupled itself to the transport library; one domain error is the seam that lets you swap `requests` for `httpx` or a real SDK without touching a panel; (c) the API's status codes are a **contract you already designed** — `422` means the *dashboard* sent something invalid (a bug in the caller, worth a distinct message), `500` is the byte cap tripping, `502` is BigQuery upstream, `404` on `/runs` is the routine post-destroy state, not an emergency. Gotchas: `response.raise_for_status()` gives you `HTTPError` but the *useful* text is in `response.json()["detail"]` — FastAPI puts it there; and `requests` treats a 404 as a perfectly successful HTTP round-trip, so `raise_for_status()` (or an explicit check) is the only thing standing between you and a `KeyError` three frames later. **Verify:** pytest against a fake transport (`monkeypatch` `requests.get`, or `responses`-style stub — no live API in tests, same rule as Day 15's `dependency_overrides`): success path, connection refused → `DashboardError`, 502 → `DashboardError` carrying the detail, and a **timeout is actually passed** (assert the kwarg — a test that doesn't check it lets the parameter be deleted silently; mutation discipline, Day 15 s5). *Commit: `add dashboard api client with error translation`.*

2. **The first panel — trending, and the date the API insists on.** 🧑‍💻 `dashboard/app.py`: `st.set_page_config`, a title, a **sidebar** with `st.date_input("Date", value=date(2024, 1, 1))` and a `limit` slider (`1..100`, matching the API's `le=100` — the UI should not be able to construct a 422), then a trending panel calling `fetch_trending(day, limit)` and rendering `st.dataframe` + `st.bar_chart` of stars by `repo_name`. Wrap the call in `try/except DashboardError` → `st.error(str(e))` and `st.stop()`. The teaching: Day 15 s2 made `?date=` **required on purpose** (a literal default would serve plausible stale 200s) — that decision now shows its bill at the consumer, which must always supply one; defaulting the *picker* to `2024-01-01` is the right place for the default, because it's a UI affordance the user can see and change, not a server-side assumption they can't. Second: mirroring the API's bounds in the widget is **defense in depth, not duplication** — the widget prevents the bad request, the API's `ge`/`le` still rejects it if the widget is ever wrong, and neither is allowed to be the only check. Gotchas: only `2024-01-01` has data, so pick any other date **once, deliberately**, and make sure what you see is a stated "No trending data for 2024-01-02" and not an empty chart that reads as "zero activity" — an empty state that looks like a data state is the UI version of dropping rows silently. `st.bar_chart` wants the label as the index; a raw list-of-dicts will chart the wrong column or the row number. **Verify:** panel renders; **three-way check** — the rank-1 repo shown in the browser == the rank-1 row from `curl.exe`/`docs` == the rank-1 row from `bq` against `trending_repos_daily`; changing the date to an empty day shows the stated empty state; killing `uvicorn` shows the error banner, not a traceback. *Commit: `add trending dashboard view`.*

Charted naïvely, one bar is the chart. The tempting fix — filter `Unknown` out of the visualization — is **the INNER-join lie relocated to the presentation layer**: the number a viewer reads would silently exclude 98.7% of reality. 4. **The pipeline status panel — the platform watching itself.** 🧑‍💻 A panel calling `fetch_runs(limit=10)`: for the newest run show `run_id`, `logical_date`, `recorded_at`, the counts (`raw_rows` / `hour_rows` / `quarantine_rows` / `residual_rows`), and the per-task states; below it, the run history table. The teaching is **tri-state rendering**: `verdict` is `bool | None` (Day 15 s5 serves NULL honestly rather than coercing), so the panel has **three** states — pass / fail / **unknown** — and collapsing `null` into either one is a lie the operator would act on. Same for `errors[]`: the API returns malformed rows in-band precisely so they can't vanish, so the panel must surface them (`st.warning`) — an `errors[]` you don't render is an in-band channel you've turned into `/dev/null`. And `404` from `/runs` means the metadata table doesn't exist, which after a nightly `terraform destroy` is **routine**, not an outage: it deserves `st.info("No pipeline runs recorded yet")`, not a red banner. Gotcha: the table will hold only the run(s) seeded since the last destroy — do not read "one row" as a bug. **Verify:** the seeded run renders with its real counts (**180,387 raw** / **180,386 hour** / **0 quarantine** / **1 residual** — the canonical GE identity); task states listed; deliberately point the client at a project where the table is absent (or rename the constant for one run) to see the 404 path render as info; confirm `bq ls -j` shows **no query job** for the panel — the free path survives the extra hop. *Commit: `add pipeline status panel from runs endpoint`.*

5. **Two caches, stacked — and the arithmetic nobody does.** 🧑‍💻 Wrap the four client calls in `@st.cache_data(ttl=300)` and add a sidebar **Refresh** button calling `st.cache_data.clear()`. The teaching: Streamlit re-runs the script on **every** widget interaction, so without this, dragging the limit slider fires four HTTP requests and (past the API's TTL) four billed BigQuery query jobs — the re-run model turns a UI fidget into a bill. Then the honest part, which is a decisions.md entry: **there are now two TTLs in series** — Streamlit's 300 s in front of the API's 300 s — so worst-case staleness is **their sum (up to ~10 minutes), not 300 s**, and the Refresh button clears only the **near** cache; the API will happily re-serve its own cached rows behind it. Name the fix without building it: a single invalidation key derived from the latest `run_id` in `pipeline_run_metadata` (the same honest key Day 15 named), propagated as a cache-busting parameter, collapses both layers to one truth. Also flag: `st.cache_data` is keyed on the function's **arguments**, so caching a function that reads a global date instead of taking it as a parameter caches the wrong thing forever — pass everything in. **Verify:** with the API stopped after a first successful load, re-interacting still renders (served from Streamlit's cache — see it, then reason about whether that's a feature or a lie); the uvicorn access log shows **no new request** on a cached re-run and exactly one on Refresh; a date change is a distinct cache key → new request. *Commit: `add dashboard caching with explicit refresh`.*

## Constraints & conventions
- 🧱 **One lesson: a dashboard is an API consumer, not a second warehouse client.** CI, auth, deploy, drill-down endpoints → named, deferred.
- 🚫 **No `google.cloud` import and no SQL string anywhere under `dashboard/`** — `grep -rn "bigquery\|SELECT" dashboard/` returning a hit is a failed day, full stop. **No new gold model or dbt change** (if a panel wants data the marts don't have, that's a Day-17+ model, not a dashboard hack). **No `requests` call without a `timeout`.** **No pandas-side aggregation that recomputes what a mart already computed** — the warehouse computes, the dashboard displays; re-summing in the client is how two numbers start disagreeing.
- 🔗 **Tests:** the client is tested against a fake transport with zero live API — the same seam discipline as `dependency_overrides` (Day 15) and the pure/IO split (Day 1). Streamlit render code is not unit-tested (honest call: it's a view layer with no logic worth pinning) — **which is exactly why any real logic must live in `api_client.py` or a pure helper, not inside a panel.**
- 🔏 **PII line:** the dashboard displays `repo_name`, `actor_login` (public), counts, and pipeline metadata. Nothing from `payload` exists downstream to leak.
- 💸 **Cost shape:** the dashboard itself costs nothing — it makes HTTP calls. Its *bill* is the API's query jobs, and the re-run model is the amplifier: **step 5's cache is a cost control, not a nicety.** `/runs` stays free (`list_rows`). `maximum_bytes_billed=100 MB` still caps every mart query. Teardown, always.
- 📐 **Lint:** ruff + black over `dashboard/`; type hints on every signature; comments carry *why* (the timeout value, error translation at the boundary, the `Unknown` presentation call, `—`-not-`0`, stacked TTLs).
- 📦 **Five commits, imperative mood:** `add dashboard api client with error translation` · `add trending dashboard view` · `add language momentum and leaderboard views` · `add pipeline status panel from runs endpoint` · `add dashboard caching with explicit refresh` · plus the docs commit.
- 🔐 **Backlog carried:** auth + rate limiting + pinned deps remain the pre-deploy gate — a dashboard does **not** count as authentication, and adding a second localhost consumer changes nothing about exposure. ## Learning goals
1. **Why the consumer goes through the API, not around it** — the API is where parameterization, the byte cap, pagination determinism, and caching are enforced; a second client is a second unguarded path to the warehouse *and* a second copy of SQL that will drift. Being able to name the one case where direct-to-warehouse wins (single consumer, no API) is what makes it a decision instead of dogma.
2. **Error translation at a boundary** — the UI must never see a transport exception; one domain error type is what keeps the presentation layer swappable and the failure messages human. The general form: every layer converts foreign failures into its own vocabulary.
3. **Timeouts as a correctness property, not a tuning knob** — a request with no timeout is an unbounded hang with no error to page on; `requests` defaults to none, so "did you set a timeout" is a real code-review question.
4. **Empty, NULL, and unknown are three distinct states, and each must look like itself** — a blank chart reading as "zero activity," a NULL delta rendered `0`, and a `null` verdict shown green are all the same bug as an INNER join dropping rows: information destroyed at the last mile, after the pipeline worked hard to preserve it.
5. **Caches in series multiply staleness** — worst case is the *sum* of the TTLs, and a refresh button only clears the layer it owns. The honest invalidation key for batch-updated marts is the pipeline's own run metadata; TTLs are the approximation you accept when you haven't wired it.
6. **CORS is a browser policy, not an API protection** — it constrains which web origins' JavaScript may *read* a response; it authenticates nobody and stops no server-side client (like Streamlit, or `curl`). Knowing when it applies — and that it is never a substitute for auth or rate limiting — is a common interview separator.
7. **Dependency isolation as a repeatable judgment call** — dbt got an image, Streamlit gets a venv, and the reasoning is identical (heavy transitive deps near `protobuf`/`pyarrow` next to `google-cloud-*`); choosing the *cheapest* isolation that avoids a security regression (not the most impressive one) is the engineering.
