# Day 10 — The center of the star: the incremental `fact_events` + the first mart (`trending_repos_daily`)

> **Phase 2 · Week 5.** Picks up from Day 9 (the **dimension ring** is complete — `dim_event_type`, `dim_repo`, `dim_actor`, `dim_date`, all tables off `ref('stg_events')` with surrogate/natural keys and passing `not_null`/`unique` tests; `dbt build` PASS=28). The dims are FK *targets* with nothing yet pointing at them. Today we build the **center of the star** — the fact — and finally light up the **`relationships`** tests those surrogate keys were built to receive, then stand up the **first gold mart** so the layer actually serves a product feature.
>
> **Scope call (read first):** today is **`fact_events` (incremental) + `relationships` tests + one mart (`trending_repos_daily`)** — no more. The other two marts (`language_momentum`, `contributor_leaderboard`) wait for Day 11, and the dbt-in-Airflow gate + Great Expectations wait for Phase 2 Week 5's back half. The fact earns a full day on its own because **incremental materialization** is the single densest lesson in dbt: `is_incremental()`, a `unique_key`, the `MERGE`, a watermark, late-arriving rows, `--full-refresh`, and the BigQuery cost shape of all of it. Building the fact *and* three marts in one day would bury that lesson under a big unreviewable diff — the exact anti-pattern the project avoids.
>
> **The incremental-strategy decision (made up front, stated honestly):** `fact_events` uses **`incremental_strategy='merge'` with `unique_key='event_id'`**, *not* `insert_overwrite`. The honest tradeoff: for **immutable, append-only** GitHub events, `insert_overwrite` by `event_date` partition would scan fewer bytes (it replaces whole partitions, the direct heir of Day 5's `table$YYYYMMDDHH` decorator + Day 6's dynamic-partition-overwrite). We deliberately choose `merge` anyway, for two reasons: (1) it exercises the **`unique_key` upsert** machinery that is the heart of the lesson, and (2) it gives **row-grain idempotency** — a reappearing `event_id` in an overlapping reload is *upserted*, never duplicated — which is the cleanest possible expression of the project's hard idempotency rule. We keep `partition_by=event_date` **anyway** so BigQuery can prune the MERGE's scan (see Step 2). `insert_overwrite` is the documented cheaper alternative for when the row-grain upsert isn't needed.

## Focus
Build **`fact_events`** as an **incremental** model (`merge`/`unique_key='event_id'`, `partition_by=event_date`) off `ref('stg_events')`: one row per event (grain = `event_id`, the **degenerate** key), carrying the four **FKs** to the dims — `repo_sk`, `actor_sk`, `date_key`, `event_type` — computed **deterministically** (regenerate the md5 surrogate from the natural key; no dimension join needed). Add the **`relationships`** tests (each FK → its dim PK) plus `unique`/`not_null` on the grain, then build the first mart **`trending_repos_daily`** (stars = `WatchEvent` count per repo per day, joined to `dim_repo`). Get **`dbt build` green end-to-end**, prove incremental **idempotency** via a second run, regenerate lineage docs, lint clean.

## Build steps
Execute in order; one small commit per step. Steps 0 and 4 are setup/verify, not commits.

0. **Recreate infra, repopulate silver, `dbt deps`, confirm the Day 9 baseline.** `terraform -chdir=terraform apply`; confirm `.env` + ADC. Land the bronze hour (`python -c "from ingestion.ingest import ingest_hour; ingest_hour('2024-01-01', 15)"` — **not** `run.py`), bring up the Spark stack and `spark-submit spark/silver_events.py 2024-01-01 15`, then `python -m transform.load_silver` — so `devpulse_silver.silver_events` carries the **180,386-row** hour. Bring up the dbt image, run **`dbt deps`**, then a `dbt build` to confirm the four dims + `stg_events` are still green (PASS=28) before we add the fact. *(Setup, not a commit.)*

1. **`fact_events` — the incremental fact (its own commit).** `models/marts/fact_events.sql`.
   - **Config block** (per-model, not in `dbt_project.yml`): `materialized='incremental'`, `incremental_strategy='merge'`, `unique_key='event_id'`, `partition_by={'field': 'event_date', 'data_type': 'date'}`, `on_schema_change='fail'` (a new/dropped column should **break loudly**, not silently drift). *(Optional `cluster_by=['event_type']` for read-side pruning in the marts — note it, decide whether to include.)*
   - **Columns** (lean — a fact holds keys + measures, not wide attributes):
     - `event_id` — the **degenerate dimension** + the `unique_key`.
     - `repo_sk = {{ dbt_utils.generate_surrogate_key(['repo_id']) }}`, `actor_sk = {{ dbt_utils.generate_surrogate_key(['actor_id']) }}` — the FKs, **regenerated deterministically** from the natural keys. This is the elegant consequence of a *deterministic* md5 surrogate: the fact does **not** need to join `dim_repo`/`dim_actor` to look the key up — same input → same hash → the `relationships` test proves it lands. (The classic ETL alternative is a **surrogate-key lookup join** against the dim; you'd need that if the dim minted *sequential* surrogate ids, or to physically enforce referential integrity / handle **early-arriving facts** — a fact referencing a dim member not yet loaded. Both our tables come from the same `stg_events`, so the dims are complete relative to the fact; the join buys nothing here. State the tradeoff, don't hide it.)
     - `date_key = cast(format_date('%Y%m%d', event_date) as int64)` — the FK to `dim_date`, matching its **smart key** exactly (same deterministic derivation).
     - `event_type` — the FK to `dim_event_type` (whose PK is the **natural** key, so the fact just carries `event_type` straight through).
     - `event_date` (the partition column) + `created_at` (the incremental **watermark**).
   - **The incremental filter** — the one beat that makes it incremental:
     ```
     {% if is_incremental() %}
     where created_at >= (
         select timestamp_sub(max(created_at), interval 1 hour) from {{ this }}
     )
     {% endif %}
     ```
     On the **first** run `{{ this }}` doesn't exist → the guard is skipped → full build. On **later** runs it selects only a trailing window; the `MERGE` then upserts on `event_id`, so re-processing that window **can't** create duplicates (row-grain idempotency). The **1-hour lookback** (not a strict `> max`) is the **late-arriving-rows** hedge: events that showed up *after* we'd already advanced the watermark get re-swept and merged. Bigger lookback = safer against lateness, more bytes scanned each run — the standing tradeoff. *(Over our single static hour the filter mostly proves the mechanism; note it's exercised, not stress-tested — same honesty as Day 9's "no rename appears over one hour.")*
   - **Factless, on purpose.** A GitHub event has no natural additive quantity (no amount, no duration) — so there is **no measure column**; the marts express volume as `COUNT(*)`. The classic Kimball "add a constant `1 as event_count`" convenience is noted and **skipped** (a plain `COUNT(*)` is clearer and costs nothing extra). *Commit (`fact_events.sql` only — the `.yml`/tests land in Step 2 so the relationships lesson reads as its own beat, or fold them together if you prefer one commit; keep it reviewable).* 

2. **`relationships` + grain tests — the payoff (its own commit).** `models/marts/fact_events.yml`.
   - **Grain:** `not_null` + `unique` on `event_id` (proves one-row-per-event survived the merge).
   - **Referential integrity — the point of the surrogate keys:** a `relationships` test on **each FK** → its dim PK:
     - `repo_sk` → `ref('dim_repo')` `.repo_sk`
     - `actor_sk` → `ref('dim_actor')` `.actor_sk`
     - `date_key` → `ref('dim_date')` `.date_key`
     - `event_type` → `ref('dim_event_type')` `.event_type`
     - Plus `not_null` on all four FKs (a NULL FK is an orphan fact row — fail loudly).
   - **dbt-1.11 nesting:** generic-test args nest under `arguments:` (the same deprecation fixed for `accepted_values` on Day 9). So a relationships test is:
     ```
     - relationships:
         arguments:
           to: ref('dim_repo')
           field: repo_sk
     ```
     Confirm the compiled test SQL under `target/compiled/` is a left-join-anti pattern (`... where dim.pk is null`). These tests are why `dim_date` had to be **gap-free** (Day 9): a WatchEvent on a day missing from `dim_date` would fail `relationships` here — the calendar completeness now has teeth. *Commit (`fact_events.yml`).*

3. **`trending_repos_daily` — the first mart (its own commit).** `models/marts/trending_repos_daily.sql`, **table** (marts default). The first model that actually **joins the fact to a dim and aggregates** — the shape every serving query takes.
   - **Grain:** one row per `(repo, day)`.
   - **Logic:** from `fact_events`, keep `event_type = 'WatchEvent'` (**GitHub's "star"** is the Watch event — the canonical trending signal), join `dim_repo` on `repo_sk` for `repo_name`, group by `(date_key, repo_sk, repo_id, repo_name)`, `stars = count(*)`. Optionally add `total_events = count(*)` over *all* event types (a second CTE) and a `rank() over (partition by date_key order by stars desc)` for the leaderboard shape. **Read `ref('fact_events')`/`ref('dim_repo')` — never the raw source** (a mart reaching past the fact to `stg_events`/`silver` is a bug: it bypasses the star and the incremental fact).
   - **Over one hour "daily" is degenerate** (one date, one hour of events) — the *shape* is what's exercised, not a real multi-day trend. Note it (like Day 9's Type-1 caveat); it comes alive once the backfill expands. *Commit (`trending_repos_daily.sql` + a `.yml` with `not_null`/`unique` on the `(date_key, repo_sk)` grain — a `unique` on a **surrogate of the two**, `generate_surrogate_key(['date_key','repo_sk'])`, or `dbt_utils.unique_combination_of_columns`).* 

4. **`dbt build` green + prove incremental idempotency + docs.** `dbt build` (full run — first build, fact is full). Then **run `dbt build` (or `dbt run -s fact_events`) a *second* time in the same session** and confirm `fact_events` takes the **incremental** path (the log says `MERGE` / "incremental", not "create or replace table") and the row count is **steady** (180,386 → still 180,386 — the merge upserted, didn't append). This second run **is** the idempotency proof for the fact, the direct analog of Day 5's partition-scoped re-load and Day 7's full-chain re-run. Spot-check integrity: `COUNT(*)` on `fact_events` = the `stg_events` row count; every `relationships` test green. `dbt docs generate` — the lineage now **fans in**: `stg_events → fact_events` with `dim_repo/dim_actor/dim_date/dim_event_type` feeding the `relationships` edges, and `fact_events + dim_repo → trending_repos_daily`. *(Setup/verify, not a commit.)*

5. **Lint + `--full-refresh` note + cost shape + cleanup + status.** `sqlfluff lint` the new models clean (confirm the templater resolves the `{% if is_incremental() %}` block + the `dbt_utils` macro calls — the Jinja may need a scoped `noqa` like Day 9's). Document two operational beats in the day log: (1) **`dbt build --full-refresh`** rebuilds the fact from scratch — the escape hatch when the incremental logic or schema changes (`on_schema_change='fail'` will *force* this on a column change); (2) **cost shape** — `merge` scans the **target** partitions it touches, and `partition_by=event_date` is what lets BigQuery **prune** that scan to the affected day(s) rather than the whole table (the concrete reason we kept partitioning even under `merge`); the marts re-scan the fact on every rebuild. Then **`docker compose down`** (both stacks) + **`terraform -chdir=terraform destroy`**. Update `CLAUDE.md` **Current status / Next up** (→ Day 11: `language_momentum` + `contributor_leaderboard` marts) and log the Day 10 decisions in `decisions.md` (merge-vs-insert_overwrite, deterministic-regenerate-vs-join for FKs, factless fact, the lookback watermark). *Commit (lint + docs/status updates).* 

## Constraints & conventions
- 🧱 **One fact + one mart only — the other two marts are Day 11.** Today completes the star's **center** and lights up the `relationships` ring; the remaining marts + the DAG gate + Great Expectations are deferred. Building all three marts now buries the incremental lesson in a big diff.
- 🔁 **Incremental via `merge` + `unique_key='event_id'` — the row-grain idempotency choice.** Stated tradeoff: `insert_overwrite` by partition would scan less for immutable events (Day 5/6 lineage), but `merge` exercises the `unique_key` upsert and dedupes a reappearing `event_id`. Keep `partition_by=event_date` regardless so BQ prunes the MERGE scan. Don't oversell — write the honest comparison in `decisions.md`.
- 🔑 **FKs are the surrogate keys, regenerated deterministically — no dim join.** `repo_sk`/`actor_sk` come from `generate_surrogate_key` over the fact's own natural keys (same md5 → same value as the dim); `date_key` from the same `YYYYMMDD` derivation; `event_type` passes straight through (natural PK of its dim). The **surrogate-key-lookup join** is the noted alternative (needed for sequential surrogates / early-arriving facts) — not needed here since both come from `stg_events`.
- 🔗 **Every FK carries `relationships` + `not_null`; the grain carries `unique`.** This is the deferred Day 9 payoff — fact FK → dim PK integrity, enforced as tests that **fail loudly**. `dim_date` being gap-free (Day 9) is what makes its `relationships` test pass; the completeness now has teeth.
- 🧾 **Factless fact — measures live as `COUNT(*)` in the marts.** No natural additive measure on a GitHub event; the classic constant-`1` counter is noted and skipped. `event_id` is the **degenerate dimension**.
- 🔌 **`ref()` only, and marts read the fact — never the source.** `fact_events` reads `ref('stg_events')`; the mart reads `ref('fact_events')`/`ref('dim_repo')`. A mart reaching back to `stg_events`/`silver_events` bypasses the star + the incremental fact — a bug.
- 🏗️ **Fact = incremental (per-model config); marts = table (the default).** The incremental config is a **per-model** block, not a `dbt_project.yml` default (only the fact is incremental). Marts stay tables — stable, fast, what the API/dashboard read.
- 🧠 **Incremental state lives in BigQuery, not git.** First build after a `destroy` is full; the incremental path only exists once `{{ this }}` does. Idempotency is proven with **two builds in one session**, and `--full-refresh` is the rebuild escape hatch.
- 🔏 **PII watch, still.** The fact carries `actor_id`/`actor_sk` only — **no email**, no reach back to bronze/`payload`. `dim_actor` already drew this line (Day 9); the fact does not widen it.
- 💸 **Cost: `merge` scans target partitions → `partition_by` prunes it; marts re-scan the fact.** Trivial over one 180k-row hour; note the at-scale shape (lookback window × partitions touched) rather than paying for it. Per-query `maximum_bytes_billed` cap is still the Phase-3 plan. `terraform destroy` + `docker compose down` at session end.
- 📐 **SQL conventions.** `sqlfluff` (BigQuery dialect, dbt templater); `fact_` prefix; models documented in `.yml` (descriptions feed the docs site). Confirm the linter handles the `is_incremental()` Jinja + the `dbt_utils` calls (scoped `noqa` if needed, Day 9 style).
- 📦 **Small, reviewable commits & understand-don't-rubber-stamp.** One commit per concept (`add incremental fact_events`, `add relationships + grain tests on fact_events`, `add trending_repos_daily mart`, `lint + docs/status`). Want the *why* for merge-vs-insert_overwrite, regenerate-vs-join FKs, factless-vs-measure, and the lookback watermark before each lands.

## Learning goals
1. **Incremental materialization** — `materialized='incremental'`, the `{% if is_incremental() %}` guard, `{{ this }}`, why the first build is full and later builds are incremental, and `--full-refresh` as the rebuild escape hatch.
2. **`merge` vs `insert_overwrite` on BigQuery** — `unique_key` upsert (row-grain idempotency, dedupes a reappearing key) vs whole-partition replace (cheaper for immutable data); why `partition_by` still matters under `merge` (scan pruning); the honest case for each.
3. **The watermark + late-arriving rows** — a `created_at` high-water mark to limit reprocessing, and the **lookback window** as the hedge against events that arrive after the watermark advanced; the safety-vs-bytes tradeoff.
4. **Fact-table design** — transaction grain (one row per event), the **degenerate dimension** (`event_id`), a **factless** fact (measures as `COUNT(*)`), and why a fact holds keys + measures, not wide attributes.
5. **Deterministic surrogate FKs vs surrogate-key-lookup joins** — regenerating the md5 key from the natural key (no join, because the hash is a pure function) vs the classic ETL dimension lookup, and when the join is actually required (sequential surrogates, early-arriving facts, physical RI).
6. **`relationships` tests = referential integrity as code** — fact FK → dim PK as a compiled anti-join, why they came online *with* the fact, and how `dim_date`'s Day-9 gap-freeness is what lets its `relationships` test pass.
