## Focus
Stand up a **native BigQuery load job** that reads the hour's silver **Parquet** from GCS into a new HOUR-partitioned `silver_events` table, **partition-scoped** via a `$YYYYMMDDHH` decorator (Day 5's rule, one layer up). Then **replace the `transform` (Counter) task** in `devpulse_ingest` with two tasks — **run Spark** (`silver_transform`) then **load to BQ** (`load_silver`) — decide and implement **how Airflow triggers the Spark container**, **delete the `Counter` path**, **drop `max_active_runs=1`**, and prove the whole `wait_for_archive >> ingest >> silver_transform >> load_silver` chain green and idempotent for one hour.

## Build steps
Execute in order; one small commit per step. Steps 1, 6 are setup/verify, not commits.

1. **Recreate infra + repopulate bronze *and* silver.** Day 6 tore the cloud down. `terraform -chdir=terraform apply`; confirm `.env` + ADC. Land one bronze hour (`python run.py`, or just `ingest_hour`), then **run the Day 6 Spark job** (`docker compose -f spark/docker-compose.yaml up -d`, then `spark-submit silver_events.py 2024-01-01 15`) so there is **silver Parquet to load**. Confirm both the bronze object and the silver `event_date=2024-01-01/event_hour=15/` partition exist in GCS. *(Setup, not a commit.)*

2. **Add the BQ `silver_events` table + a Parquet load job.** New module (e.g. `transform/load_silver.py`, mirroring `event_counts.py`'s shape). Two decisions to make explicit here:
   - **Partition column = `created_at` (TIMESTAMP), not `event_hour`.** Spark's `partitionBy` **strips `event_date`/`event_hour` out of the Parquet files** — they live only in the GCS *path*. `created_at` **is** in the files, so HOUR-partition the BQ table on it (continues Day 5's HOUR grain). *(Decision: if we also want `event_date`/`event_hour` as columns, recover them with BQ **hive partitioning options** — `source_uri_prefix` + `AUTO` mode — pointed at the `events/` root. Recommended: skip hive options today, derive any date/hour dims in dbt staging (Phase 2); load the one hour's prefix directly. Keeps the load narrow.)*
   - **Native BQ load job (`load_table_from_uri`, `SOURCE_FORMAT=PARQUET`), not the spark-bigquery connector.** The load job is **free**, decouples compute (Spark) from load (BQ), and reuses the exact partition-decorator idempotency pattern from Day 5. *(Rejected alt: write straight from Spark via the spark-bigquery connector — another jar, couples the write to the Spark run, and loses the clean "Parquet in the lake is the contract; BQ is a load target" separation. The connector is the right call for streaming/huge writes; for a free hourly batch load it's overkill.)*
   - Create the table idempotently (`create_table(exists_ok=True)`) with an **explicit schema** matching the silver columns (`event_id`, `event_type`, `actor_id`, `actor_login`, `repo_id`, `repo_name`, `public`, `created_at`); HOUR `time_partitioning` on `created_at`. *Commit (the loader module + schema).*

3. **Load silver Parquet → BQ, partition-scoped & idempotent.** In `load_silver(date, hour)`: point the load at `gs://{silver}/events/event_date={date}/event_hour={hour}/*.parquet` (⚠ **`event_hour` is an unpadded int** in the path — `event_hour=15`, *not* `15`-zero-padded like the bronze `hour=15` dir; verify the exact dir name in GCS), into `silver_events${YYYYMMDDHH}` with `WRITE_TRUNCATE`. The decorator means **every loaded row's `created_at` must fall in that hour partition or BQ rejects the job** — that's the *loud-failure* guard, and it surfaces any cross-hour `created_at` boundary rows (note + explain any rejects, don't loosen the decorator). **Verify:** `SELECT COUNT(*)` and a `GROUP BY event_type` over the partition reconcile to the **Day 6 silver count (180,386)** for that hour; re-run the load and confirm the partition is **replaced, not doubled**. *Commit.*

4. **Wire Spark into Airflow: replace `transform` with `silver_transform >> load_silver`.** The central decision of the day — **how does an Airflow task launch the Spark job**, given Spark lives in a separate stack?
   - **Recommended: `DockerOperator`.** Airflow launches the `devpulse-spark` image with `spark-submit silver_events.py {{ ds }} {{ hour }}` as its command, via the **mounted Docker socket** (`/var/run/docker.sock`). Cleanest local shape: keeps the heavy Spark JVM out of the Airflow worker, reuses the Day 6 image unchanged, one container per run. 🔐 **Security flag:** mounting the host Docker socket gives the Airflow worker **root-equivalent control of the host** — acceptable for a local single-node dev box, but call it out as a real surface and note that at scale you'd use the **KubernetesPodOperator** / a Spark-on-k8s submit or Dataproc operator instead (no socket).
   - *Rejected alts:* **`BashOperator` + `docker compose run`** (works, but shells out to compose — brittle arg passing, same socket need, less observable than DockerOperator's typed XCom/logs); **`SparkSubmitOperator`** (designed for submitting to a *cluster* via a Spark connection — needs the Spark client baked into the Airflow image; mismatched with single-node Docker); **merge Spark into the Airflow image** (one stack, no socket — but a multi-GB image and couples the two lifecycles; loses the clean per-layer separation Day 6 built).
   - Build it: `silver_transform` (DockerOperator → `spark-submit`) then `load_silver` (PythonOperator → `transform/load_silver.py`, same logical-date derivation as the existing `_ingest`/`_transform`). Pass `date`/`hour` from the data interval, exactly like the current tasks. *Commit (the new tasks + wiring, Counter still present but unreferenced).*

5. **Retire the `Counter` + drop `max_active_runs=1`.** Cut the DAG over: `wait_for_archive >> ingest >> silver_transform >> load_silver`; remove the old `transform`/`_transform` task and its import of `load_event_counts`. **Delete (or clearly deprecate) `transform/event_counts.py`** — it's superseded; note in the commit that `hourly_event_counts` was the Day 3–5 placeholder. **Remove `max_active_runs=1`**: Day 5 added it *only* because the in-memory `Counter` couldn't survive backfill fan-out; Spark spills to disk and is the thing that lifts that cap, so removing it is the **whole point** of the swap (say so in the commit). *(Caveat: with both stacks local, parallel runs now mean parallel 4g Spark drivers — fine for a couple, but note the practical local ceiling; the cap is gone *architecturally*, RAM is the new *operational* limit.)* *Commit.*

6. **Run the full DAG end-to-end + prove idempotency across both layers.** Trigger the `2024-01-01 15:00` run (or a clear+rerun). Confirm all four tasks green, the silver `event_date=…/event_hour=15/` Parquet exists, **and** `silver_events$2024010115` in BQ matches. Then **re-run the whole DAG for the same hour** and confirm: (a) the GCS partition is replaced not doubled (Day 6's dynamic-overwrite), (b) the BQ partition row count is steady (Day 5's decorator) — **both idempotency mechanisms holding in one chain**. *(Setup/verify, not a commit.)*

7. **Test + cost hygiene + status + tick the box.** Add a focused unit test for the **new** logic — the partition-decorator / path derivation in `load_silver` (e.g. `2024-01-01`,`15` → prefix `event_hour=15` and decorator `$2024010115`); the Spark transform already has its Day 6 test. *(Small commit.)* Then **`docker compose down`** (both stacks) + **`terraform -chdir=terraform destroy`**. Update `CLAUDE.md` **Current status / Next up** (→ Day 8), log the Day 7 decisions in `decisions.md`, and **finally tick the Phase 1 PySpark milestone box** — it's done now: the Spark silver job is wired into Airflow, loading BigQuery, with the `Counter` retired. *Commit.*

## Constraints & conventions
- 🔁 **Idempotency is mandatory (hard rule) — now in *two* places at once.** The same hour re-run must replace, not duplicate, **both** the silver Parquet (dynamic partition overwrite, Day 6) **and** the BQ partition (`$YYYYMMDDHH` + `WRITE_TRUNCATE`, Day 5). Step 6 proves the whole chain, not each half in isolation.
- 🎯 **Partition grain = load grain, again.** Load **one hour's** Parquet prefix into **one** BQ hour partition via the decorator. Never load the `events/` root into the table unscoped — that's the static-overwrite / whole-table-truncate trap, two layers deep now.
- 📐 **Explicit schema on the BQ load, not autodetect.** Parquet carries its own schema, but pin the BQ table schema explicitly (same contract principle as the Spark `StructType` and the Day 3 `mode="REQUIRED"`): autodetect would silently absorb drift. Decide the `created_at` partition column deliberately given `partitionBy` stripped the path columns.
- 🧱 **The cap comes off because Spark replaces the bound, not because we got brave.** `max_active_runs=1` was a *correctness* fence around an in-memory transform (Day 5). It's safe to remove **only** because the distributed job is now the transform. Frame it that way; don't drop it as an unrelated cleanup.
- 🔐 **Docker socket = host root (new surface today).** If `silver_transform` uses DockerOperator/`docker.sock`, that's the Airflow worker getting root-equivalent host control. Acceptable for local dev; flag it explicitly and name the at-scale fix (KubernetesPodOperator / Spark-on-k8s / Dataproc — no socket). No secrets in DAG code or git.
- 🔐 **Still personal ADC; SA + silver-bucket grant still deferred.** Terraform already has the pipeline SA's **BQ silver dataset** editor grant (`pipeline_silver_editor`) and **bronze bucket** objectAdmin — but the **silver *bucket* object grant** for the SA is still missing, and we run as **personal ADC**. Not blocking today (ADC has both). The SA-impersonation switch + silver-bucket grant remain on the security backlog; note they'd land together.
- 🔏 **PII watch (carries).** Silver still excludes `payload` (author emails). The BQ load only moves the existing narrow silver columns — don't widen the schema to pull `payload`/emails in. Payload explosion + hashing is Phase 2 staging.
- 💸 **Cost hygiene.** The **Parquet load job is free** (no bytes-billed; loads aren't query jobs). Costs today are bronze read + tiny silver Parquet + silver table storage. Spark stays **local in Docker**. `docker compose down` (both stacks) + `terraform destroy` at session end.
- 📦 **Small, reviewable commits & understand-don't-rubber-stamp.** One step per commit (`add silver_events bq table + parquet loader`, `load silver parquet to bigquery`, `trigger spark from airflow via DockerOperator`, `retire counter transform, drop max_active_runs`). Want the *why* for created_at-vs-event_hour partitioning, load-job-vs-connector, and DockerOperator-vs-alternatives before each lands.

## Learning goals
1. **BigQuery load jobs from Parquet** — load-vs-query (why a load job is free and isn't bytes-billed), Parquet's embedded schema vs. an explicit BQ schema as the contract, and `WRITE_TRUNCATE` + the `$YYYYMMDDHH` decorator as the warehouse idempotency primitive (recapped from Day 5, now from a Parquet source).
2. **The `partitionBy`-strips-the-columns gotcha** — why `event_date`/`event_hour` vanish from the Parquet files and live only in the GCS path, what hive-partitioning options do to recover them, and why partitioning the BQ table on the in-file `created_at` is the clean move.
3. **Cross-stack orchestration** — how Airflow drives a job in a *separate* container: DockerOperator + the Docker socket vs. SparkSubmitOperator vs. a merged image, the tradeoffs, and the socket-as-host-root security cost (and its k8s/Dataproc at-scale fix).
4. **Retiring a placeholder safely (the thin-slice "swap")** — building the real layer beside the working one, then cutting over and deleting the stand-in; why `max_active_runs=1` was a bound that the distributed transform *earns the right* to remove, not arbitrary cleanup.
5. **Idempotency composed across layers** — dynamic partition overwrite (lake) *and* the partition decorator (warehouse) holding **in a single DAG run**, so the whole bronze→silver→BQ chain is replayable for any hour.
6. **Decoupling compute from load** — Parquet-in-the-lake as the durable contract and BQ as a swappable load target (load job vs. spark-bigquery connector), and why that separation is worth a second hop.
