## Environment notes (Windows 10 · PowerShell · Docker Desktop)
Carries from Day 7. dbt runs in **its own Docker image** (`dbt/Dockerfile` + `dbt/docker-compose.yaml`), invoked `docker compose -f dbt/docker-compose.yaml run --rm dbt <cmd>`. You still bring up Spark in Step 0 to *produce* a silver hour to model against, but the modeling itself is a third, lightweight container.
- **dbt runs in Docker, NOT the pipeline `.venv` — decided Day 8 after a real conflict.** Installing `dbt-bigquery` into the shared `.venv` downgraded `google-cloud-storage` and broke `black` (dbt pins `pathspec` low). dbt's dependency tree is too heavy to cohabit with the pipeline runtime, so it gets an isolated image — which is also the form it takes later (an Airflow DockerOperator, like Spark, and a CI image). The `.venv` was recreated clean. *(See decisions.md.)*
- Wiring `dbt build` into the Airflow DAG as a **test gate** is **Phase 2 Week 5**, not today — a deliberate deferral, flagged so it's a conscious choice, not a forgotten gap.
- **Keyless auth, still personal ADC.** dbt authenticates to BigQuery via `method: oauth` over the same Application Default Credentials the rest of the project uses (`gcloud auth application-default login`). **No SA key**, consistent with the Day 2 ADC decision. The pipeline-SA-impersonation switch + silver-bucket grant remain on the security backlog (not touched today).
- **`profiles.yml` is gitignored.** It holds project id, dataset, and location — no secrets under OAuth, but keep it out of git on principle (it's host/user-specific config, and the convention is "connection config never in the repo"). `dbt debug` is the connectivity smoke test.
- GCP project **`devpulse-dp2622`**; silver dataset **`devpulse_silver`** (the dbt *source*); gold dataset **`devpulse_gold`** (new today — the dbt *target*). BigQuery **location must match** across datasets or dbt's cross-dataset reads fail — confirm `devpulse_gold` is created in the **same region** as `devpulse_silver`.
- **`curl.exe`** (not `curl`); no `gzip`/`head`/`wc` in PowerShell — inspect BQ with `bq`/Python, dbt with its own CLI output and the compiled SQL under `target/`.
- The daily `terraform destroy` empties the buckets and (if added to Terraform) drops `devpulse_gold`. To have data to model against you must **re-ingest a bronze hour → re-run Spark silver → `load_silver`** before `dbt run`. **Order of operations matters:** bronze hour → silver Parquet → BQ `silver_events` → dbt. dbt has nothing to read until `silver_events` is populated.

## Build steps
Execute in order; one small commit per step. Steps 0 and 6 are setup/verify, not commits.

0. **Recreate infra, repopulate silver, stand up the dbt image.** ✅ *(done)* Day 7 tore the cloud down. `terraform -chdir=terraform apply`; confirm `.env` + ADC. Land one bronze hour (`python -c "from ingestion.ingest import ingest_hour; ingest_hour('2024-01-01', 15)"` — **not** `run.py`, which still wires the deprecated Counter), bring up the Spark stack and `spark-submit silver_events.py 2024-01-01 15` (full path `/opt/spark/bin/spark-submit` if PATH-not-found), then `python -m transform.load_silver` — so `devpulse_silver.silver_events` has the **180,386-row** hour to model against. **Build the dbt image** (`docker compose -f dbt/docker-compose.yaml build`) and confirm `... run --rm dbt --version` shows the bigquery plugin. *(Setup, not a commit; `dbt/Dockerfile` + compose **are** committed — they're the new tooling.)*

1. **Provision the `devpulse_gold` dataset (IaC, not by hand).** Gold output gets its **own dataset**, separate from `devpulse_silver` — clean layer separation, independent access/cost boundaries, and it mirrors the medallion split in the warehouse itself (silver = pipeline-owned typed events; gold = dbt-owned business models). **Decision:** add `devpulse_gold` to **Terraform** (`google_bigquery_dataset`, same region as silver) rather than letting dbt create it ad hoc — infra is IaC in this project, and a dbt-created dataset would be an untracked resource `terraform destroy` wouldn't clean up. Grant the pipeline SA dataEditor on it for later (it's free to declare now even while we run as ADC). `terraform apply`; confirm the empty dataset exists in the right region. *Commit (the Terraform change).*

2. **Scaffold the dbt project.** Under `dbt/` (alongside the already-built `Dockerfile`/compose): `dbt_project.yml` (name `devpulse`, model paths, a `models/` tree of `staging/` + `marts/` — or `staging/`+`dims/`+`facts/`+`marts/`; pick one and note it), and a **gitignored `profiles.yml`** with a `devpulse` profile → BigQuery, `method: oauth`, `project: devpulse-dp2622`, `dataset: devpulse_gold`, `location: us-central1` (matches silver's region), sane `threads`. Set **default materializations per layer** in `dbt_project.yml`: `staging: view`, everything else `table` for now (incremental comes with `fact_events`, a later day). Add `dbt/target/`, `dbt/dbt_packages/`, `dbt/logs/`, and `dbt/profiles.yml` to `.gitignore`. **`docker compose -f dbt/docker-compose.yaml run --rm dbt debug` must pass** (the auth/connectivity smoke test — the dbt analog of Day 6 proving the GCS connector before any transform). *Commit (project scaffold; `dbt debug` green).*

3. **Declare `silver_events` as a dbt `source`.** `models/staging/_sources.yml`: a `silver` source over dataset `devpulse_silver`, table `silver_events`, with column descriptions (lineage + docs payoff). This is the **dbt seam between layers** — staging reads `{{ source('silver','silver_events') }}`, **never** a hardcoded table reference, so dbt owns the dependency graph. *(Optional, recommended: add a `freshness` block on the source — `loaded_at_field: created_at` — so `dbt source freshness` can later assert silver isn't stale. Note it even if you don't wire it into a gate today.)* *Commit (source declaration).*

4. **First staging model: `stg_events` (a view).** `models/staging/stg_events.sql` — the 1:1 cleanup/standardize layer over the source: select + rename to house style, **recover `event_date` and `event_hour`** by deriving them from `created_at` (`DATE(created_at)`, `EXTRACT(HOUR FROM created_at)` — the columns Spark's `partitionBy` stripped out of the Parquet and that the BQ load therefore doesn't carry), and any light, safe casts. **No business logic, no joins, no aggregation** — staging is plumbing; dims/marts are where modeling happens. Materialized as a **view** (set by the layer default — cheap, zero storage, always-fresh against silver). `dbt run -s stg_events` builds it; eyeball the compiled SQL under `target/compiled/`. *Commit (`stg_events` view).*

5. **First dimension: `dim_event_type` + tests.** `models/marts/dim_event_type.sql` (or `models/dims/` per your Step 2 layout): `SELECT DISTINCT event_type FROM {{ ref('stg_events') }}`, materialized as a **table**. Chosen as the **first dim deliberately** — it's the smallest, has a **natural key** (`event_type` itself), and exercises the full staging→dim→test loop **without** the surrogate-key generation that `dim_repo`/`dim_actor` will need (that pattern lands Day 9). Add a `schema.yml`: `not_null` + `unique` on `dim_event_type.event_type`, `accepted_values` (the documented GitHub event types — `PushEvent`, `PullRequestEvent`, `WatchEvent`, …; let it **fail loudly** if silver carries a type you didn't list, which is exactly the data-quality signal we want), and `not_null`/`unique` on `stg_events.event_id`. *Commit (`dim_event_type` + schema tests).*

6. **`dbt build` green end-to-end + docs.** `dbt build` (runs **models then tests** in dependency order — the one command that proves the slice). All models build, all tests pass. Then `dbt docs generate` (+ `dbt docs serve` to eyeball the **lineage graph**: `silver_events` → `stg_events` → `dim_event_type`) — the lineage DAG is the visible Phase-2 payoff and a portfolio highlight. *(Setup/verify, not a commit.)*

7. **Lint + cost note + cleanup + status.** Add a **`sqlfluff`** config (`.sqlfluff` with the BigQuery dialect + the dbt templater) per the SQL conventions, and lint the models clean. Note the **BigQuery cost shape**: `dbt build` is **query jobs** (bytes-billed, unlike Day 7's free Parquet *load* job) — trivial over one 180k-row hour now, but flag that at backfill scale staging-as-view re-scans silver on every downstream build, so partition-pruning discipline (and later, incremental) matters. Update `CLAUDE.md` **Current status / Next up** (→ Day 9: `dim_repo`/`dim_actor` with surrogate keys + `dim_date`), log the Day 8 decisions in `decisions.md` (gold-dataset-via-Terraform, OAuth/keyless dbt, staging-as-view, dbt-local-not-in-Airflow-yet). *Commit (lint config + docs/status updates).*

## Constraints & conventions
- 🧱 **Thin slice, not the whole star schema.** Today is the **dbt bootstrap + one vertical sliver**. `dim_repo`/`dim_actor` (surrogate keys), `dim_date`, the **incremental `fact_events`**, and all three marts are **explicitly deferred** to Days 9–10. Building more today trades the "prove the toolchain green first" discipline for a big unreviewable diff — the exact anti-pattern the project avoids.
- 🔌 **`source()`/`ref()` only — never hardcode a table.** Staging reads `source('silver','silver_events')`; dims read `ref('stg_events')`. This is what gives dbt the dependency graph, lineage, and safe rebuilds. A literal `devpulse_silver.silver_events` in a model is a bug.
- 🏗️ **Materialization is a deliberate per-layer choice.** Staging = **view** (cheap, no storage, always fresh). Dims = **table** (stable, fast to read). Fact = **incremental** — *later*. Say *why* each is what it is; don't let dbt defaults decide silently.
- 🔐 **Keyless dbt auth (carries the ADC rule).** `method: oauth` over ADC — **no SA key minted**, consistent with Day 2. `profiles.yml` is **gitignored**; no credentials, project secrets, or `target/` artifacts in git.
- 🧰 **IaC owns the gold dataset.** `devpulse_gold` is provisioned by **Terraform**, not created ad hoc by dbt — so it's tracked, region-pinned, and torn down cleanly. Same region as `devpulse_silver` or cross-dataset reads break.
- 🚦 **Tests gate the build; pipelines fail loudly.** `dbt build` (not bare `dbt run`) so tests run with the models. `accepted_values` on `event_type` should **fail** on an unlisted type — that's the data-quality signal, not noise to suppress. Per conventions, all gold models carry tests.
- 🔏 **PII watch (carries).** Silver already excludes `payload`/author emails — don't reach back to bronze or widen the model to pull them into gold. If a later dim needs actor identity, that's `actor_id`/`actor_login`, not email.
- 💸 **Cost: `dbt build` is bytes-billed (unlike the Day 7 load job).** Trivial over one hour now; note the at-scale shape (view re-scans, partition pruning, incremental) rather than paying for it. `terraform destroy` + `docker compose down` at session end. The per-query `maximum_bytes_billed` cap is still the Phase-3 plan.
- 📐 **SQL conventions.** `sqlfluff` (BigQuery dialect, dbt templater); `stg_` prefix for staging, `dim_` for dimensions. Models documented in `schema.yml` (descriptions feed the docs site).
- 📦 **Small, reviewable commits & understand-don't-rubber-stamp.** One step per commit (`add devpulse_gold dataset (terraform)`, `scaffold dbt project + bigquery profile`, `declare silver_events dbt source`, `add stg_events staging view`, `add dim_event_type + schema tests`, `add sqlfluff config + dbt docs`). Want the *why* for gold-as-its-own-dataset, OAuth-vs-keyfile, view-vs-table materialization, and source()-vs-hardcoded before each lands.

## Learning goals
1. **dbt project anatomy** — `dbt_project.yml` vs `profiles.yml`, the `models/` tree, and how `source()`/`ref()` build the dependency DAG (so dbt — not you — orders and rebuilds the models).
2. **Keyless warehouse auth from dbt** — `method: oauth` over ADC, why a project standardizes on no-SA-keys, and what `dbt debug` actually checks.
3. **Materialization strategy & its cost shape** — view vs table (vs the incremental that's coming), what each does to storage and to bytes-scanned on rebuild, and why staging is a view but dims are tables.
4. **Sources & the layer seam** — declaring silver as a dbt source, `loaded_at_field`/freshness as a staleness assertion, and why staging is the only place that touches the source.
5. **Staging discipline** — the 1:1 "clean and standardize, no business logic" layer, and recovering the `event_date`/`event_hour` columns `partitionBy` stripped (the gotcha that chains all the way from Day 6 → Day 7 → here).
6. **Tests as a build gate + lineage docs** — `dbt build` running models *and* tests, `accepted_values` as a loud data-quality signal, and the auto-generated lineage graph as the visible Phase-2 deliverable.
7. **Why gold is its own Terraform-managed dataset** — warehouse-level medallion separation, and keeping infra in IaC instead of letting the transform tool create untracked resources.
