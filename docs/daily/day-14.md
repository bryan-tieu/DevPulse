>
> **The day's central decision, stated up front and honestly:** *when is a retry rational, and when is it noise?* Day 12/13 proved retries can't fix data — the validator re-failed identically three times while the human who could fix it was uninformed. So: **infra-shaped tasks keep `retries=2`** (sensor flake, network blip — idempotency makes re-runs free), **data-gate tasks get `retries=0`** (`validate_silver`, `dbt_build` — a failing check is deterministic; the correct response is a page, immediately). Alerts fire from `on_failure_callback` — which Airflow invokes only **after retries exhaust**, so the policy and the callback compose: infra flakes self-heal silently, data failures page on the first and only attempt. *Rejected:* alerting on every retry (page fatigue — the fastest way to teach humans to ignore alerts); email/SMTP (no local mail path; webhook is the modern shape anyway); parsing task logs for metadata (**logs are for humans, artifacts are for machines** — the validator will emit a JSON summary instead, closing Day 13's log-vs-exit-code arc properly). **Simplification flagged loudly:** the webhook URL in `.env` is local-dev posture; production = secrets backend (Airflow Connections on a real backend, K8s secrets). And the metadata table is written by a task *inside* the DAG it observes — honest for one DAG; at scale observability moves out-of-band (listeners/OpenLineage) so the observer can't die with the patient.

## Constraints & conventions
- 🧱 **One lesson: the pipeline speaks.** Alert (push, on real failure) + metadata (pull, every run) — both fed by the same verdict booleans Day 13 built. Slack apps, SLAs, listeners/OpenLineage, dashboards → named, deferred.
- 🚫 **No alert on retries** (page fatigue kills alerting programs faster than missed pages do); **no streaming inserts** (free load jobs exist); **no log parsing** for metadata (artifacts are for machines); **no webhook URL anywhere near git** — `.env` only, and the payload carries ids + counts, never event data.
- 🔗 **Both paths or it isn't done:** green run (no page, metadata row) *and* red run (one attempt, page received, downstream dead, metadata row anyway). An alert that has never fired and a logger that has never survived a failure are both decorations.
- 🔏 **PII line:** alert payloads and metadata rows carry dag/task ids, states, and counts — no `actor_login`, no event content, nothing from `payload`. The webhook endpoint is third-party; treat everything sent there as public.
- 💸 **Cost shape:** metadata writes are free load jobs; the table is tiny and unpartitioned (say so — partitioning a KB-scale table is cargo-culting). Red-path drill re-runs the chain twice (Spark local, loads free). Teardown, always.
- 🔐 **Backlog carried:** webhook secret → secrets backend (Phase 4); pipeline-SA impersonation still pending; per-query `maximum_bytes_billed` still Phase 3.
- 📐 **Lint:** ruff + black; type hints on every new signature; pure/I-O split enforced in both the callback and the logger (it's what the pytests cover); comments carry *why* (timeout on the POST, `retries=0`, `all_done`, missing-summary NULLs).
- 📦 **Four commits, imperative mood:** `add failure alert webhook callback` · `route alerts vs retries by failure class` · `emit machine-readable run summary from validator` · `add run metadata logging task` · plus the docs commit.

## Learning goals
1. **Retry policy is a routing decision, not a number** — retries exist for stochastic failures (infra flakes made free by idempotency); data failures are deterministic, so retrying them only delays the page — being able to argue `retries=0` *as the correct setting* for a gate is the interview answer.
2. **Callbacks compose with retries** — `on_failure_callback` fires only after the retry budget exhausts, which is exactly what makes "retry infra silently, page data immediately" expressible with two knobs.
3. **Logs are for humans, artifacts are for machines** — parsing log lines couples consumers to formatting; a structured summary derived from the same verdict booleans gives every consumer (operator, alert, metadata table, Day 15+ API) one source of truth.
4. **`trigger_rule="all_done"` and the observer problem** — the default `all_success` is why gates stop pipelines, and the override is why observers must opt out of being stopped; a metadata logger that dies with the failure it should record is worse than none.
5. **Free load jobs vs streaming inserts** — the same cost discipline as Day 7's load-not-query decision, applied to telemetry; at this scale the pennies don't matter but the habit does.
6. **Page fatigue as a first-class design constraint** — alerting on every retry trains humans to ignore alerts; the scarcest resource in an alerting system is the recipient's trust.
7. **In-band vs out-of-band observability** — a task inside the DAG can record that DAG honestly, until the scheduler itself is the failure; the production shape (listeners, OpenLineage, external observers) survives what it observes.
