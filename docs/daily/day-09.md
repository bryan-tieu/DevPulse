## Build steps
Execute in order; one small commit per step. Steps 0 and 5 are setup/verify, not commits.

0. **Recreate infra, repopulate silver, stand up dbt + `dbt deps`.** Day 8 tore the cloud down. `terraform -chdir=terraform apply`; confirm `.env` + ADC. Land one bronze hour (`python -c "from ingestion.ingest import ingest_hour; ingest_hour('2024-01-01', 15)"` — **not** `run.py`, which still wires the deprecated Counter), bring up the Spark stack and `spark-submit silver_events.py 2024-01-01 15` (full path `/opt/spark/bin/spark-submit` if PATH-not-found), then `python -m transform.load_silver` — so `devpulse_silver.silver_events` carries the **180,386-row** hour. Bring up the dbt image and run **`dbt deps`** (fetches `dbt_utils`); a `dbt build` here should still be green on the Day 8 models before we add anything. *(Setup, not a commit.)*

1. **Add the `dbt_utils` package.** Create `dbt/packages.yml` pinning `dbt-labs/dbt_utils` to a compatible version range (pin a real range, don't float — reproducible builds). Run `dbt deps` in-container and confirm it lands under `dbt/dbt_packages/` (gitignored). This is the dbt analog of `requirements.txt`: the **manifest is committed, the fetched code is not**. We need it for two macros today: **`generate_surrogate_key`** (a deterministic hash key) and **`date_spine`** (a generated date range). *Commit (`packages.yml` only — verify `dbt_packages/` is ignored).*

2. **`dim_repo` + `dim_actor` — surrogate keys, Type-1 latest-wins (one commit).** Both follow the **same pattern**, so they land together — the diffs are near-identical; the only distinct content is `dim_actor`'s PII boundary. Each is a **table** (marts default), one row per entity.
   - **`dim_repo`** (`models/marts/dim_repo.sql`): `{{ dbt_utils.generate_surrogate_key(['repo_id']) }} as repo_sk` — an md5 hash, the dimension's PK and the single-column FK `fact_events` will reference — and keep `repo_id` (the **natural/business key**) as a column. Take the **latest** `repo_name` via `QUALIFY ROW_NUMBER() OVER (PARTITION BY repo_id ORDER BY created_at DESC) = 1` (BigQuery's `QUALIFY` filters a window function without a subquery). **Surrogate tradeoff (be honest):** a single-source clean integer `repo_id` would work fine as the key *as-is* — we adopt the Kimball surrogate to practise the pattern and give the fact a uniform key shape; don't pretend it's strictly necessary. **SCD:** this is **Type-1 (overwrite)** — current name only, no history; **Type-2** (versioned rows, valid-from/valid-to) is the deferred production alternative. *(Over one hour no rename appears — the pattern is exercised, not stress-tested.)*
   - **`dim_actor`** (`models/marts/dim_actor.sql`): the same shape over `actor_id`/`actor_login` — `generate_surrogate_key(['actor_id']) as actor_sk`, keep `actor_id` (natural key), latest `actor_login` via the same `QUALIFY` Type-1 rule. **🔏 PII checkpoint — the one new beat:** the actor dimension is exactly where someone would be tempted to enrich with an email. **Don't.** Silver already excludes `payload`/author emails; gold identity stays `actor_id`/`actor_login` (a public username) **only** — don't reach back to bronze to widen it.
   - **Tests** (`dim_repo.yml`, `dim_actor.yml`): `not_null` + `unique` on **both** the surrogate and the natural key of each (a surrogate collision *or* a natural-key dupe both fail loudly), `not_null` on `repo_name`/`actor_login`. *Commit (`dim_repo` + `dim_actor` + tests — one commit).*

3. **`dim_date` — a *generated* calendar, not `DISTINCT event_date`.** `models/marts/dim_date.sql`, table. The key insight: a date dimension is **conformed and pre-generated**, *independent of the fact data*. Building it from `SELECT DISTINCT event_date FROM stg_events` would give a **gappy** calendar (an hour/day with no events = a missing row), which breaks date-range joins and rollups. Instead generate a **contiguous** spine with `dbt_utils.date_spine` over a fixed range (e.g. all of 2024, covering the backfill window), then enrich each day with calendar attributes: `year`, `month`, `day`, `day_of_week`, `day_name`, `week`, `quarter`, `is_weekend`. **Key choice — the smart key:** use a `YYYYMMDD` **integer `date_key`** (e.g. `EXTRACT`-derived) as the PK, *not* `generate_surrogate_key`. The date dimension is the **deliberate exception** to "surrogate keys should be meaningless" — a readable, sortable integer date key is a long-standing Kimball convention (aids partition pruning + human-readable fact rows). `dim_date.yml`: `not_null`/`unique` on `date_key` and `full_date`, range/`accepted_values` sanity tests on `month`/`day_of_week` if useful. *Commit (`dim_date` + tests).*

4. **`dbt build` green end-to-end + docs.** `dbt build` (models **then** tests, in DAG order). All four dims + `stg_events` build, all tests pass. Verify grain quickly: `dim_repo`/`dim_actor` row counts equal the `COUNT(DISTINCT repo_id)`/`COUNT(DISTINCT actor_id)` over the hour (no fan-out from the rename logic), and `dim_date` is gap-free across its range. `dbt docs generate` (+ `dbt docs serve` to eyeball the lineage: `silver_events → stg_events → {dim_event_type, dim_repo, dim_actor}` plus the standalone `dim_date`). *(Setup/verify, not a commit.)*

5. **Lint + cost note + cleanup + status.** `sqlfluff lint` the new models clean (the dbt templater resolves `ref()`/`source()`/the `dbt_utils` macros — confirm the macro calls don't trip the linter). Note the **cost shape**: the dims are now **tables** (`dbt build` issues bytes-billed query jobs + writes storage, unlike a view's 0-byte definition) — trivial over one hour, but at backfill scale each dim rebuild re-scans `stg_events` (which re-scans silver), the standing argument for partition pruning and the coming incremental fact. Then **`docker compose down`** (both stacks) + **`terraform -chdir=terraform destroy`**. Update `CLAUDE.md` **Current status / Next up** (→ Day 10: the **incremental `fact_events`** + first mart), and log the Day 9 decisions in `decisions.md` (dbt_utils package, surrogate-vs-natural key, Type-1 latest-wins via QUALIFY, generated date_spine + smart key). *Commit (lint + docs/status updates).*

## Constraints & conventions
- 🧱 **Three dims only — `fact_events` and the marts are deferred to Day 10.** Today completes the **dimension** ring of the star. The incremental fact is its own meaty day (per Day 8); starting it now without the relationships-test target is half a lesson and a big unreviewable diff — the exact anti-pattern the project avoids.
- 🔑 **Surrogate keys via `dbt_utils.generate_surrogate_key`, with the tradeoff stated.** `repo_sk`/`actor_sk` are md5 hashes; the natural key (`repo_id`/`actor_id`) stays as a column. Be honest: a clean single-source integer id would serve as the key today — we adopt the surrogate to practise the Kimball pattern and give the fact a uniform single-column FK. Don't oversell necessity.
- 🕰️ **Type-1 (overwrite) dims now; Type-2 (history) noted, not built.** `repo_name`/`actor_login` are mutable; we keep only the **latest** (`QUALIFY ROW_NUMBER() … ORDER BY created_at DESC = 1`). SCD Type-2 (valid-from/valid-to versioned rows) is the production alternative — flag it, defer it.
- 📅 **`dim_date` is generated, not derived.** `date_spine` over a fixed range → a **gap-free** conformed calendar independent of the events, with a **smart `YYYYMMDD` integer key** (the deliberate exception to "meaningless surrogate"). Never `SELECT DISTINCT event_date` — that yields a gappy calendar.
- 🔌 **`source()`/`ref()` only — never hardcode a table.** Dims read `ref('stg_events')`; a literal `devpulse_silver.silver_events` (or `devpulse_gold.stg_events`) in a model is a bug — it breaks the DAG and lineage.
- 🏗️ **Dims = table (the marts default).** Stable, fast to read, the FK targets the fact will join. No per-model materialization override needed today; the incremental override lands with `fact_events`.
- 🚦 **Every dim carries tests; `dbt build` gates.** `not_null`/`unique` on **both** the surrogate and the natural key of each dim (a surrogate collision or a natural-key dupe both fail loudly). **`relationships` tests are deferred to Day 10** — they link the fact's FKs to these dims' PKs, and there's no fact yet.
- 🔏 **PII watch (the `dim_actor` checkpoint).** Actor identity in gold is `actor_id`/`actor_login` (public username) **only** — never email. Don't reach back to bronze or widen the dim to pull `payload`/emails in. Silver already dropped them; gold keeps it that way.
- 💸 **Cost: dims are tables → bytes-billed query jobs + storage.** Trivial over one 180k-row hour; note the at-scale shape (each dim rebuild re-scans `stg_events`→silver) rather than paying for it. `terraform destroy` + `docker compose down` at session end. Per-query `maximum_bytes_billed` cap is still the Phase-3 plan.
- 📐 **SQL conventions.** `sqlfluff` (BigQuery dialect, dbt templater); `dim_` prefix; models documented in `schema.yml` (descriptions feed the docs site). Confirm the linter handles the `dbt_utils` macro calls.
- 📦 **Small, reviewable commits & understand-don't-rubber-stamp.** One commit per concept (`add dbt_utils package`, `add dim_repo + dim_actor (surrogate keys, type-1)`, `add dim_date (generated calendar)`, `lint + docs/status`). Want the *why* for surrogate-vs-natural, Type-1-vs-Type-2, generated-vs-distinct date dim, and smart-key-vs-hash before each lands.

## Learning goals
1. **dbt packages** — `packages.yml` vs the fetched `dbt_packages/` (the manifest-committed / code-ignored split, like `requirements.txt` vs `.venv`), `dbt deps`, the dbt package hub, and what `dbt_utils` gives you.
2. **Surrogate vs. natural keys** — `generate_surrogate_key` (a deterministic md5 hash), why a star schema prefers a uniform single-column surrogate PK/FK, and the honest case where a clean single-source integer natural key would do just as well.
3. **Slowly-changing dimensions** — Type-1 (overwrite, latest-wins) via `QUALIFY ROW_NUMBER()` vs Type-2 (versioned history with valid-from/valid-to); why `repo_name`/`actor_login` are mutable and `repo_id`/`actor_id` are stable.
4. **The conformed date dimension** — `date_spine` to generate a **gap-free** calendar independent of the fact data (vs the gappy `DISTINCT event_date` trap), and the `YYYYMMDD` **smart key** as the deliberate, principled exception to "surrogate keys are meaningless."
5. **Dimension grain & BigQuery `QUALIFY`** — "one row per business entity," verifying grain against `COUNT(DISTINCT …)`, and `QUALIFY` as BQ sugar for filtering a window function without a wrapping subquery.
6. **Building toward the fact** — these dims are precisely the FK targets `fact_events` will reference; why the `relationships` tests (fact→dim) come online with the fact on Day 10, and how the lineage DAG now fans out from `stg_events`.
