# Day 12 — Gate the pipeline: `dbt build` as an Airflow task, and prove the gate actually gates

> **Phase 2 · Week 5 (back half — quality).** Picks up from Day 11 (the gold layer is **complete**: `repo_languages` seed + `language_momentum` + `contributor_leaderboard` joined `trending_repos_daily`; `dbt build` **PASS=69**; reconciled 180,386 fact rows / 163,953 allowlisted contributions). But every one of those 69 tests runs only **when a human remembers to type `dbt build`**. The pipeline itself — `devpulse_ingest`, hourly — happily lands new silver data and stops there: gold goes stale and *nothing checks anything* unless we do. Today the tests stop being a dev-loop habit and become a **pipeline gate**: a `dbt_build` task appended to the DAG, so every ingested hour rebuilds gold **and a failing test fails the run, loudly, in the orchestrator**.
>
> **Scope call (read first):** today is **one DockerOperator task + the proof it gates** — no more. **Great Expectations moves to Day 13** (it's a whole new tool; the "Day 12" bundle in the old status was two lessons — split per the one-day-one-lesson rule). **Failure alerting and run-metadata logging are deferred with it** (they belong with GE's counted, alerting checks). Also OUT: astronomer-cosmos (rendering the dbt DAG as individual Airflow tasks — real tool, heavier dependency; our thin slice is one task), `--select`-based partial builds, and any schedule redesign. The chain simply grows one link: `wait_for_archive >> ingest >> silver_transform >> load_silver >> dbt_build`.
>
> **The day's central decision, stated up front and honestly:** *where* should dbt run from Airflow? We choose a **DockerOperator launching the existing `devpulse-dbt` image** — the exact pattern Day 7 used for Spark, deliberately: per-tool images are our unit of orchestration, and Day 8 already proved dbt's dependency tree cannot cohabit with anything (it broke the `.venv`; installing dbt into the *Airflow* image would be the same mistake with worse blast radius). *Rejected:* dbt-in-the-Airflow-image (dependency cohabitation, the Day 8 lesson ignored); `BashOperator + docker compose run` (same socket, worse observability — the Day 7 analysis holds unchanged); cosmos (right at scale for per-model retries + observability, overkill for a 9-model project — named as the production-grade step, not used). **Simplification flagged loudly:** wiring `dbt build` *into the hourly ingest DAG* couples the modeling cadence to the ingest cadence — every hour rebuilds and re-tests *all* of gold (fine at 69 tests/30 s; at scale you'd decouple: a separate transform DAG triggered by Airflow Datasets, or dbt orchestrated by cosmos with state-based selection). We accept the coupling because the *lesson* — tests gate the pipeline — lives in the wiring, and the decoupling changes the trigger, not the gate. And the day is not done when the task goes green: **a gate that has never failed is unproven** — step 3 deliberately breaks a test and watches the DAG go red before we call it a gate.

## Environment notes (Windows 10 · PowerShell · Docker Desktop)
Carries from Day 11 unchanged — plus the Airflow stack, dormant since Day 7, comes back up.
- **Keyless auth, still personal ADC.** No SA key, ever. The scheduler already injects `HOST_ADC` (host path to `application_default_credentials.json`) for exactly this purpose — the dbt task binds it `:ro` the same way Spark does. 🔐 Pipeline-SA impersonation + silver-bucket grant: still on the security backlog, untouched today.
- **Stacks needed:** Airflow (`docker compose -f airflow/docker-compose.yaml up -d`, webserver on :8080) **and** the `devpulse-dbt` image must exist on the **host daemon** (`docker compose -f dbt/docker-compose.yaml build` if `docker image ls` doesn't show it) — DockerOperator launches sibling containers by *image name*; it does not read compose files. Spark image likewise (the chain runs it). Keep the DAG **paused**; trigger runs manually pinned to the canonical hour, exactly as Day 7 did.
- **The daily `terraform destroy` emptied everything** → `/start-session` chain first (bronze → Spark → `load_silver` → `dbt deps` → baseline `dbt build` **PASS=69**). dbt has nothing to test until `silver_events` carries the 180,386-row hour.
- **New today — reading DockerOperator against a compose file it will never see.** The dbt container has so far been *described by compose* (`dbt/docker-compose.yaml`: env, mounts, entrypoint). DockerOperator replicates that contract in Python: what compose spells `volumes:`/`environment:`/`entrypoint:` the operator spells `mounts=[Mount(...)]`/`environment={...}`/`command=`. Translating between the two — and knowing what lives in the **image** (`DBT_PROFILES_DIR`, `WORKDIR` — baked by the Dockerfile, carries automatically) versus only in **compose** (the `entrypoint: ["dbt"]` — does **not** carry!) — is the day's hands-on skill.
- `curl.exe` not `curl`; Docker bind **sources resolve on the host daemon** — hence `HOST_PROJECT_DIR`, never the scheduler's `/opt/...` paths (the Day 7 trap; it *will* bite again if forgotten, as a silently empty mount).

Bring up Airflow; confirm `devpulse_ingest` parses with no import errors and shows the current 4-task chain; confirm `devpulse-dbt` exists on the host daemon. *(Setup, not a commit.)*

1. **The `dbt_build` task — DockerOperator, mirroring the compose contract.** 🧑‍💻 **This is the day's core lesson; Bryan writes it.** In `airflow/dags/devpulse_ingest.py`: a `DBT_ENV` dict and a `DockerOperator(task_id="dbt_build", image="devpulse-dbt", ...)`, chained `load_silver_task >> dbt_build`. Open `dbt/docker-compose.yaml` side-by-side as the spec, and `silver_transform` as the in-DAG template. The traps to expect (find them yourself first; the hint ladder is open):
   - **The entrypoint is compose-only.** `dbt/Dockerfile` bakes no `ENTRYPOINT` — compose adds `entrypoint: ["dbt"]` at run time. DockerOperator won't. A `command="build"` therefore dies with "executable not found"; the command must be the full **`dbt build`** (WORKDIR + `DBT_PROFILES_DIR` *are* baked into the image, so no `--project-dir` gymnastics needed).
   - **The project mount is read-write** — the *only* RW mount in the DAG. Spark's code mount is `:ro`; dbt must write `target/` and `logs/` into the mounted project dir. Source = `f"{HOST_PROJECT_DIR}/dbt"` (host path, Day 7 rule).
   - **ADC mount:** `HOST_ADC` → `/opt/dbt/gcp/adc.json`, `read_only=True`, with `GOOGLE_APPLICATION_CREDENTIALS` pointing at it — mirror the compose env exactly (plus `GOOGLE_CLOUD_PROJECT` from the scheduler's env, like `SPARK_ENV` does).
   - **Windows/DooD boilerplate carries:** `docker_url="unix://var/run/docker.sock"`, `mount_tmp_dir=False`, `auto_remove="success"` — same three as Spark, same reasons.
   - **Retries are safe here — say why in a comment:** `default_args` gives this task `retries=2`; that's only *correct* because every layer it rebuilds is idempotent (incremental fact = `merge` on `unique_key`; marts/dims = full `CREATE OR REPLACE`). A non-idempotent build step with auto-retry would corrupt on flake — the Day 1 rule is what makes the retry policy free.
   - While in the file (🤖 fold into the same commit): DAG `tags` gain `"phase2"`, description mentions the gate.
   - **Verify:** DAG re-parses clean (`airflow dags list-import-errors` empty). *Commit: `add dbt build gate task to devpulse_ingest`.*

2. **Green path — the full 5-task chain on the canonical hour.** 🧑‍💻 Trigger a manual run pinned to the canonical hour (Day 7 mechanics). Watch the chain: sensor → ingest (skips, blob exists) → Spark (dynamic overwrite, idempotent) → `load_silver` (`WRITE_TRUNCATE` into `$2024010115`) → **`dbt_build`**. Success = all green **and** the `dbt_build` task log ends `PASS=69 WARN=0 ERROR=0`. Anything that breaks here, Bryan debugs first (observation → hypothesis → then hints): the likely suspects are, in order, the entrypoint trap, a scheduler-path-instead-of-host-path mount (symptom: dbt can't find `dbt_project.yml` — the mount is silently empty), and a missing env var (symptom: `Env var required but not provided` from profiles rendering). *(No commit — this is verification; the day log records the run id + PASS line.)*

3. **Red path — prove the gate gates.** 🧑‍💻 A gate that has never failed is a decoration. Temporarily break one test **in a way that mimics real bad data**: edit `dbt/seeds/repo_languages.yml` and add an `accepted_values` test on `language` whose list omits a value that exists (or tighten any existing test to be wrong) — *do not commit this*. Trigger the DAG again: `dbt build` fails the test, exits non-zero, the container exits, **the task goes red, and the run is marked failed** — downstream of the gate, nothing would run. Read the task log: dbt names the failing test and the row count; that log *is* the alert payload (the actual alerting hook is Day 13's lesson, deferred, not forgotten). Revert the sabotage, re-trigger, watch it go green — the full red→green cycle is the proof. **Bank the observation** for `decisions.md`: retries interact with real failures — the task will retry twice *even on a genuine test failure* (retries can't fix bad data; they exist for infra flakes). That's an accepted bluntness today; per-cause retry policy / alert-vs-retry routing is exactly what Day 13's alerting work refines. *(No commit — the sabotage never lands; the finding goes in the day log + `decisions.md` in step 5.)*

## Learning goals
1. **Tests-as-a-gate vs tests-as-a-habit** — the difference between a test suite that runs when a human remembers and one wired into orchestration so bad data *stops the pipeline*; why the gate belongs downstream of the load it guards.
2. **The DockerOperator pattern, generalized** — second use of per-tool-image orchestration (Spark, now dbt): compose describes the container for the dev loop, the operator replicates that contract for the scheduler, and knowing what lives in the image (WORKDIR, env baked by Dockerfile) vs the compose file (entrypoint!) is what makes the translation safe.
3. **Idempotency is what makes retries a free lunch** — `retries=2` on a build step is only sound because merge-on-unique-key and WRITE_TRUNCATE make re-execution harmless; and conversely, retries are the wrong tool for *data* failures (they can't fix a failing test — that's alerting's job, Day 13).
4. **Cadence coupling as an architectural smell** — bolting transform onto the ingest DAG works and teaches the gate, but couples "how often data lands" to "how often models rebuild"; the production shape (Dataset-triggered transform DAG, cosmos with state selection) decouples them *without changing the gate*.
5. **Negative testing infrastructure** — a gate is unproven until you've watched it fail; the red-path drill (break, observe, revert) is the same discipline as testing a backup by restoring it.
6. **The at-scale transport swap** — DockerOperator-over-socket → `KubernetesPodOperator`/managed runners: the task definition survives, only the submission transport changes; being able to name that swap precisely is the interview answer to "would you run this in production?"
