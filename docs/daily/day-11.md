>
> **The enrichment decision (made up front, stated honestly):** GH Archive event envelopes don't include the repo's language, and we deliberately **dropped `payload` from silver on Day 6** (varying shape per event type + it's where author-email **PII** lives). So there is **no path to language from the data we currently hold** — re-widening silver to fish a language out of `PullRequestEvent.payload.pull_request.base.repo.language` would re-introduce the exact shape-drift and PII surface we cut, for a field only *some* event types carry. We reject that. Instead we bring language in as **reference data via a dbt `seed`** — a committed CSV mapping `repo_id → language` — LEFT-joined in the mart, with unmatched repos bucketed as `'Unknown'` (**never dropped**). This is explicitly a **stand-in for a production enrichment pipeline**: at real scale, language comes from a scheduled **GitHub REST API** job (`GET /repos/{owner}/{repo}` → `.language`, rate-limited, cached in a `repo_metadata` table) or a repo-metadata batch source — not a hand-authored CSV. The seed teaches the *pattern* (reference data joined into a mart, no-orphan-drop handling) honestly, and swaps to the real source without touching the mart SQL (both are just `ref()`/`source()` targets). Over one hour most repos won't be in the seed, so `language_momentum` is **mostly `'Unknown'` and "momentum" over a single day is degenerate** — the mart *shape* is exercised, not a real trend, same honesty as Day 10's "daily" caveat.

1. **`repo_languages` seed — the enrichment reference source (its own commit).** `seeds/repo_languages.csv` with a header `repo_id,language` and a realistic **sample** of well-known repos and their primary language (e.g. a few hundred rows: `torvalds/linux`→C, popular Python/JS/Go repos, etc. — keyed by the **numeric `repo_id`** so the join matches the fact's key, not the mutable name). Add `seeds/repo_languages.yml`: `unique` + `not_null` on `repo_id` (a dupe would fan out the mart join), `not_null` on `language`. `dbt seed` it in-container; confirm it lands as a **table** in `devpulse_gold` and `SELECT COUNT(*)` matches the CSV rows.
   - **Why a seed, and what it is *not*:** a seed is dbt's mechanism for **small static reference data you own and version** — the CSV is committed, so the data is reviewable and reproducible, and `ref('repo_languages')` slots it into the DAG exactly like a model. It is explicitly **not** how you'd load language at scale: production replaces this CSV with a **scheduled GitHub-repos-API enrichment** landing a `repo_metadata`/`repo_language` **source table** (rate-limited, cached, incrementally refreshed) — and because the mart reads it through `ref()`/`source()`, that swap **never touches the mart SQL**. State the simplification loudly (per the project's "flag production-grade vs simplified" rule); don't pretend a 300-row CSV is the real enrichment layer.
   - **Data-type note:** BigQuery seed column types are inferred from the CSV; `repo_id` must land as **`INT64`** to join cleanly against the fact's `repo_id` — if inference makes it a string, pin the type via a `seeds:` `column_types` config in `dbt_project.yml` (a small, worth-knowing seed gotcha). *Commit (`repo_languages.csv` + `.yml` + any `column_types` config).*

2. **`language_momentum` — enrichment join + the momentum shape (its own commit).** `models/marts/language_momentum.sql`, table. Grain = one row per `(language, day)`.
   - **The enrichment join — LEFT, and no orphan drops:** the lean Day-10 fact carries `repo_sk` (the surrogate), **not** the natural `repo_id` the seed is keyed on — so route through the dim: `ref('fact_events')` **INNER JOIN** `ref('dim_repo')` on `repo_sk` (safe — the Day-10 `relationships` test proved every `repo_sk` matches, so it drops nothing) to recover `repo_id`, then **LEFT JOIN** `ref('repo_languages')` on `repo_id`, then `COALESCE(language, 'Unknown') as language`. **LEFT, not INNER** on the *seed* is the load-bearing choice: an INNER join would **silently drop every event whose repo isn't in the seed** — which over one hour is *most of them* — corrupting the totals into "only the languages we happened to seed." LEFT + `COALESCE('Unknown')` keeps every event and makes the coverage gap **visible as an `'Unknown'` bucket** rather than hiding it (the "fail loudly / never silently drop" rule applied to a join). Note honestly in a comment: over the seeded sample, `'Unknown'` dominates — that's the *coverage* of a stand-in seed, not a bug.
   - **Measures:** `event_count = count(*)` and `active_repos = count(distinct repo_sk)` per `(language, date_key)`.
   - **The momentum shape (a window function, computed in the mart):** `lag(event_count) over (partition by language order by date_key)` for the prior period, then `event_count - prev_event_count as momentum_delta` (and optionally a growth-rate / `rank() over (partition by date_key order by event_count desc)` for the day's top languages). **Over one day `lag` is NULL and "momentum" is degenerate** — there's no prior period to compare against; the mart *shape* (the period-over-period window) is what's exercised, and it comes alive once the backfill spans multiple days. Flag it exactly like Day 10's degenerate "daily."
   - **Read the fact, never the source** (a mart reaching back to `stg_events`/`silver` bypasses the star + incremental fact — a bug). The only non-fact input is the **reference seed**, joined for enrichment. *Commit (`language_momentum.sql` + a `.yml`: `unique_combination_of_columns(date_key, language)` grain guard, `relationships` `date_key → dim_date`, `not_null` on the grain + measures).*

3. **`contributor_leaderboard` — the easy mart, with the "what is a contribution" decision (its own commit).** `models/marts/contributor_leaderboard.sql`, table. Grain = one row per `(actor, day)`. Structurally a near-clone of `trending_repos_daily` (fact → dim → aggregate → `rank()`), so the diff is small and the *only* real decision is the metric definition.
   - **Logic:** from `ref('fact_events')`, join `ref('dim_actor')` on `actor_sk` for `actor_login`, group by `(date_key, actor_sk, actor_id, actor_login)`, `contributions = count(*)`, `rank() over (partition by date_key order by contributions desc) as daily_rank` (RANK, not ROW_NUMBER — ties share a rank, matching `trending_repos_daily`).
   - **The decision — not every event is a "contribution":** a leaderboard of *contributors* shouldn't count a `WatchEvent` (starring) or `ForkEvent` as a contribution. Filter the fact to a **contribution event-type allowlist** — `PushEvent`, `PullRequestEvent`, `IssuesEvent`, `PullRequestReviewEvent`, `PullRequestReviewCommentEvent`, `IssueCommentEvent`, `CommitCommentEvent`, `CreateEvent` (judgement call; state it in a comment and in `decisions.md`). **Honest tradeoff:** this counts every allowed event **equally** — a merged PR weighs the same as a single comment. Real leaderboards **weight** contributions (a `CASE` assigning points per type); we note the weighting as the deferred refinement and keep the flat count today (clearer, and the weighting is a business-rule tuning, not an engineering gap).
   - *Commit (`contributor_leaderboard.sql` + a `.yml`: `unique_combination_of_columns(date_key, actor_sk)` grain guard, `relationships` `date_key → dim_date` **and** `actor_sk → dim_actor`, `not_null` on the grain + `contributions`/`daily_rank`).*

## Constraints & conventions
- 🧱 **Two marts + one seed only — the DAG gate + Great Expectations are Day 12.** Today **completes the gold serving layer** (all three marts) and closes Phase 2's modeling milestone. The dbt-in-Airflow test gate and the GE bronze→silver gates are the back half of Phase 2 Week 5 — deferred, flagged, not forgotten.
- 🌱 **Language comes from a committed `seed`, not the event stream — the enrichment stand-in, stated honestly.** Events don't carry language and `payload` was dropped from silver (Day 6, shape + PII). The seed is small, owned, version-controlled reference data; production replaces it with a **GitHub-repos-API `repo_metadata` source** and the mart SQL doesn't change (both are `ref()`/`source()` targets). Don't oversell a 300-row CSV as the real enrichment layer — write the swap plan in `decisions.md`.
- 🚫 **Do not re-widen silver to fish out `payload.…language`.** Re-introducing `payload` for a field only some event types carry brings back the shape-drift + author-email PII we deliberately cut. Rejected — the seed is the clean answer.
- 🔗 **Enrichment join is LEFT + `COALESCE('Unknown')` — never INNER, never a silent drop.** An INNER join to a partial seed silently deletes every un-seeded event and corrupts the totals. LEFT keeps every row and surfaces the coverage gap as a visible `'Unknown'` bucket. Reconcile `SUM(event_count)` to the fact's `COUNT(*)` to prove nothing dropped.
- 🏆 **"Contribution" is a defined allowlist, counted flat — weighting is deferred.** `WatchEvent`/`ForkEvent` aren't contributions; filter the fact to the contribution event types. Every allowed event counts **equally** for now (a merged PR = a comment); per-type weighting (a `CASE`) is the noted refinement, not a gap. State the allowlist in `decisions.md`.
- 🔌 **`ref()` only, and marts read the fact — never the source.** Both marts read `ref('fact_events')` + a dim (`dim_actor`) / the seed (`repo_languages`). A mart reaching back to `stg_events`/`silver_events` bypasses the star + the incremental fact — a bug.
- 🏗️ **Marts = table (the default); the seed = a table too.** No per-model config today (the incremental override lives only on the fact). Marts stay tables — stable, fast, what the API/dashboard read; the seed materializes as a small table dbt owns.
- 🔗 **Every FK carries `relationships`; every grain carries a `unique_combination` guard.** `date_key → dim_date` on both marts; `actor_sk → dim_actor` on the leaderboard. `language_momentum` grain = `(date_key, language)`, `contributor_leaderboard` grain = `(date_key, actor_sk)` — a dupe means the `GROUP BY` leaked or a join fanned out; fail loudly.
- 🔏 **PII watch, still.** `contributor_leaderboard` carries `actor_id`/`actor_sk`/`actor_login` (public username) only — **no email**, no reach back to bronze/`payload`. `dim_actor` already drew this line (Day 9); the leaderboard doesn't widen it. The seed carries `repo_id`/`language` only — no PII.
- 💸 **Cost: both marts re-scan `fact_events` on every rebuild; the seed is ~free.** Marts don't prune — they read the whole fact and aggregate. Trivial over one 180k-row hour; the re-scan is a scale cost the upstream incremental fact already addresses. `dbt seed` loads a tiny CSV (negligible). Per-query `maximum_bytes_billed` cap is still the Phase-3 plan. `terraform destroy` + `docker compose down` at session end.
- 📐 **SQL conventions.** `sqlfluff` (BigQuery dialect, dbt templater); marts descriptively named; models documented in `.yml` (descriptions feed the docs site). Confirm the linter handles the seed `ref()`, the `lag()`/`rank()` window functions, and the `COALESCE`.
- 📦 **Small, reviewable commits & understand-don't-rubber-stamp.** One commit per concept (`add repo_languages enrichment seed`, `add language_momentum mart`, `add contributor_leaderboard mart`, `lint + docs/status`). Want the *why* for seed-vs-payload-vs-API, LEFT-vs-INNER, the contribution allowlist, and flat-vs-weighted contributions before each lands.

## Learning goals
1. **dbt seeds** — the fifth dbt object type (after source, model, test, docs): a **committed CSV** loaded as a table via `dbt seed`, for small static **reference data you own** — how it differs from a source (external, pipeline-loaded) and from `dbt_packages/` (fetched, gitignored), and why the CSV *is* committed.
2. **Enrichment / reference-data modeling** — bringing an attribute the fact doesn't carry (language) in via a lookup, the **stand-in-vs-production** honesty (seed today, GitHub-API `repo_metadata` source at scale), and why `ref()`/`source()` make that swap invisible to the mart.
3. **LEFT vs INNER on an enrichment join** — why a partial reference table demands a **LEFT** join + `COALESCE('Unknown')` (keep every fact row, surface the coverage gap) and how an INNER join **silently drops** rows and corrupts aggregates — the "never drop silently" rule at the join level, provable by a `SUM`-reconciliation.
4. **Metric definition as a modeling decision** — "what counts as a contribution" (an event-type **allowlist**, not every event) and **flat vs weighted** counting, i.e. that a mart encodes *business rules*, not just SQL, and those rules belong in `decisions.md`.
5. **Window functions in marts** — `rank()` for a leaderboard (ties shared) and `lag()` for **period-over-period momentum**, computed in the *mart* (not the fact), and why "momentum"/"daily" is a **degenerate shape** over one hour that comes alive with the multi-day backfill.
6. **Completing a dimensional serving layer** — how three marts off one conformed star (fact + shared dims + a reference seed) become the exact tables the Phase-3 API/dashboard read, and why finishing the marts *closes* Phase 2's modeling milestone and *opens* the quality-gate work (dbt-in-Airflow + Great Expectations) for Day 12.
