>
> **The day's central decision, stated up front and honestly:** *should a user-facing API query the warehouse directly?* Today: yes — the API queries BigQuery per request, because traffic is one developer, marts are KB-to-MB scale, and freshness-after-each-run matters more than milliseconds. **Rejected (and when each wins):** replicating marts into a serving DB (Postgres/Cloud SQL) — wins the moment real users arrive, because warehouses bill per query and answer in seconds while serving DBs answer in milliseconds at fixed cost; BI Engine / warehouse-native acceleration — wins for BI-tool dashboards that must stay inside BQ; pre-exported static JSON behind a CDN — wins when a small fixed query set meets big anonymous read traffic. The production-vs-simplified line, said loudly: **direct-BQ serving is the simplification** — the honest at-scale shape is "the warehouse computes, a serving store serves," and today's clean query-layer boundary (`api/queries.py`) is exactly the seam where a serving DB would slot in. Within that: values are **query parameters, never string-interpolated** (the injection line); identifiers (table names) *can't* be parameterized — they come from config constants only, never from request input, and that distinction is the interview answer; and the TTL cache is an in-process dict — honest for one worker, named as Redis / run-id-keyed invalidation at scale.

## Constraints & conventions
- 🧱 **One lesson: serving a warehouse without getting burned** — parameterize the values, cap the bytes, validate at the edge, paginate deterministically, cache deliberately. Dashboard, CI, auth, deploy → named, deferred.
- 🚫 **No f-string SQL values anywhere** — the step-1 builders are the only place SQL is assembled, and `grep -n "f\"\"\"" api/` finding a value interpolation is a step-1 failure. **No `SELECT *`** (bytes billed = columns scanned — name the columns). **No new serving infra** (no Postgres, no Redis, no BI Engine — today's scale doesn't earn them; articulating *when they'd win* is the deliverable). **No `insert`/write paths in the API** — it is read-only over gold + run metadata.
- 🔗 **Tests are the dependency-injection dividend:** builders tested pure (SQL + params, no GCP); endpoints tested via `TestClient` + `dependency_overrides` with a fake BQ client; cache tested with a fake clock. An API test that needs live GCP is a design smell today.
- 🔏 **PII line:** the API serves `repo_name`, `actor_login` (public), counts, and pipeline metadata — nothing from `payload` exists downstream to serve. `/runs` exposes op counts and task states only.
- 💸 **Cost shape:** every mart query is a billed query job (10 MB minimum applies — pennies, but the cap + cache are the habit); `/runs` is a free `list_rows`; `maximum_bytes_billed=100 MB` on every job; no `SELECT *`; teardown, always.
- 🔐 **Backlog moved and carried:** parameterized queries + `maximum_bytes_billed` **land today**; auth + rate limiting + deliberate CORS + pinned deps remain the pre-deploy gate (API stays localhost-only until then); pipeline-SA impersonation + silver-bucket grant still pending.
- 📐 **Lint:** ruff + black; type hints on every signature; pure/IO split enforced (builders, cache, row-sorting pure; client calls thin); comments carry *why* (identifiers-vs-values, ORDER BY for pagination, monotonic clock, list_rows-not-query, the 100 MB cap).

## Learning goals
1. **Parameterized queries, and the values-vs-identifiers line** — why binding values kills injection structurally (the value never enters the SQL text), why identifiers *can't* be bound in any engine, and therefore why table/column names must only ever come from config — the complete interview answer, not just "use parameters."
2. **`maximum_bytes_billed` as a per-query circuit breaker** — cost control at the job level fails an over-budget query *before* it scans; contrast with billing alerts (after the fact) and IAM caps (too blunt); the habit matters at 10 MB scale because it's unchangeable muscle memory at 10 TB scale.
3. **Serving straight from the warehouse vs a serving database** — per-query billing, seconds-latency, and concurrency limits make direct-BQ serving a small-scale simplification; being able to name the trigger (real users) and the migration seam (the query layer built today) is the production-thinking answer.
4. **Pagination in a columnar warehouse** — LIMIT/OFFSET shapes the response, not the bytes scanned; deterministic ORDER BY is what makes pages coherent at all; keyset pagination fixes deep-offset cost in serving DBs.
5. **Caching as a freshness policy** — the TTL encodes "how stale is acceptable," and the honest invalidation key for batch-updated marts is the pipeline's own run metadata; in-process vs shared cache is a deployment-topology decision, not an implementation detail.
6. **Dependency injection as the testability seam** — swapping the BQ client via `dependency_overrides` is the API-layer version of the pure/IO split the pipeline has enforced since Day 1.
7. **Free vs billed read paths** — `list_rows` tabledata reads vs query jobs, completing Day 14's free-load-jobs arc: cost-aware engineers know *which* API they're calling, not just what data comes back.
